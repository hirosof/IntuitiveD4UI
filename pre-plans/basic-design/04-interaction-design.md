## 1. 概要

本設計書では、Intuitive D4UI におけるユーザー操作とインタラクションに関する設計を定める。

### 1.1 本設計書の範囲

| 領域 | 内容 |
|------|------|
| 選択操作 | クリック選択、範囲選択、追加選択、ドリルダウン選択の操作モデル |
| ドラッグ&ドロップ | 要素の移動、コンテナ間移動、親子関係変更、ドラッグ中フィードバック |
| リサイズ・回転 | Konva.Transformer を利用した変形操作、修飾キーによる制約 |
| グループ化とコンテナの操作境界 | グループ/コンテナ間のエッジケース3件の設計 |
| Undo/Redo | コマンドパターン、スコープ、履歴管理、連続操作の結合 |
| その他の操作 | コピペ、Zオーダー変更、整列、テキスト編集、リンク設定、ロック |
| キーボードショートカット | 全操作のショートカット一覧と設計方針 |

### 1.2 前提とする設計書からの入力

本設計書は、以下の設計書で決定された事項を前提とする。

#### テーマ1（アプリケーション基盤）からの前提

- **技術スタック**: Vue 3（Composition API）+ Pinia + Konva.js + vue-konva
- **アプリケーション層の責務**: 操作コマンド（Undo/Redo対応）、選択管理、ツール状態管理
- **Konva.jsのイベントシステム**: ヒットテスト、ドラッグ&ドロップ、変形インタラクション

#### テーマ2（データ設計・永続化）からの前提

- **データモデル**: フラットマップ + 参照ID方式（`Record<string, ElementNode>`）
- **親子関係**: `parentId` / `childrenIds` で管理
- **Zオーダー**: `childrenIds` 配列の順序（先頭が最背面、末尾が最前面）
- **座標系**: ルート要素はキャンバス絶対座標、子要素は親からの相対座標
- **ElementNode 操作**: `parentId`/`childrenIds` を操作する D&D 設計、Pinia ストア上のフラットマップ操作

#### テーマ3（キャンバス・レンダリング設計）からの前提

- **3レイヤー構成**: gridLayer（静的）、elementLayer（操作対象）、overlayLayer（UI補助描画）
- **レイアウトエンジン**: calculateLayout 関数（effectiveSize パラメータ）、Vue computed で計算結果保持
- **Flex/Grid コンテナ内でのドラッグ**: `childrenIds` の順序変更で再配置（テーマ3 セクション3.7.1）
- **スマートガイド**: 同一階層要素間の辺・中心の整列ガイド（テーマ3 セクション5.4）
- **ネスト深さ制限**: 推奨8階層（警告）、上限16階層（拒否）。チェック箇所: ドロップ時・ペースト時・テンプレート展開時
- **座標変換**: ビューポート座標 ↔ キャンバス座標の変換式（テーマ3 セクション4.3）
- **テキスト編集フロー**: ダブルクリック→DOMオーバーレイ→ESC/外部クリックで確定（テーマ3 セクション2.5.3）

## 2. 選択操作

### 2.1 選択モデル

選択状態は Pinia ストアで管理する。

```typescript
interface SelectionState {
  /** 選択中の要素IDの配列 */
  selectedIds: string[];

  /** 選択コンテキスト（ドリルダウン中のコンテナ/グループID。null = ルート階層） */
  selectionContext: string | null;
}
```

| 状態 | 説明 |
|------|------|
| `selectedIds` が空 | 要素未選択。プロパティパネルは非表示またはページ情報を表示 |
| `selectedIds` が1件 | 単一選択。プロパティパネルに選択要素の情報を表示 |
| `selectedIds` が複数 | 複数選択。プロパティパネルに共通プロパティを表示 |
| `selectionContext` が null | ルート階層で操作中 |
| `selectionContext` が要素ID | そのコンテナ/グループの内部で操作中（ドリルダウン状態） |

### 2.2 クリック選択

キャンバス上の要素をクリックした場合の動作を以下に定める。

| 操作 | 動作 |
|------|------|
| 要素をクリック | 現在の `selectionContext` 階層内でヒットテストを行い、該当する最前面の要素を選択 |
| キャンバス空白をクリック | 選択解除（`selectedIds` を空にする）。`selectionContext` を null にリセット |
| ロックされた要素をクリック | 選択はできるが、ドラッグ・リサイズ・回転は不可 |
| 非表示の要素 | ヒットテスト対象外（クリックで選択できない） |

#### 2.2.1 ヒットテストの階層制限

`selectionContext` によりヒットテストの対象階層を制限する。

```
selectionContext = null（ルート階層）の場合:
  → rootElementIds に含まれる要素のみがヒットテスト対象

selectionContext = "container-A" の場合:
  → container-A の childrenIds に含まれる要素のみがヒットテスト対象
```

これにより、コンテナの内部に入っていない状態では、内部の要素を直接クリックしてもコンテナ自体が選択される。

### 2.3 範囲選択（ラバーバンド）

キャンバスの空白領域からドラッグを開始した場合、範囲選択（ラバーバンド）を行う。

| 項目 | 仕様 |
|------|------|
| 描画先レイヤー | overlayLayer |
| 表示 | 半透明の矩形（青色系、枠線付き） |
| 判定方式 | ラバーバンド矩形と要素のバウンディングボックスが交差する要素を選択 |
| 対象階層 | 現在の `selectionContext` 階層内の要素のみ |
| 修飾キー | Shift を押しながら範囲選択すると、既存の選択に追加 |

#### 2.3.1 範囲選択のフロー

```
1. キャンバス空白でマウスダウン
2. ドラッグ開始 → overlayLayer にラバーバンド矩形を描画開始
3. ドラッグ中 → マウス位置に合わせてラバーバンド矩形を更新
   → 矩形と交差する要素をリアルタイムでハイライト表示
4. マウスアップ → 交差する全要素を selectedIds に設定
   Shift 押下中: 既存の selectedIds に追加（トグル動作）
```

### 2.4 追加選択（Shift+クリック）

| 操作 | 動作 |
|------|------|
| Shift+クリック | 選択をトグル（未選択→選択に追加、選択済み→選択から除外） |

### 2.5 全選択

| 操作 | 動作 |
|------|------|
| Ctrl+A | 現在の `selectionContext` 階層内の全要素を選択 |

`selectionContext` が null の場合はルート要素を全選択、コンテナ内の場合はそのコンテナの `childrenIds` を全選択する。

### 2.6 ネスト構造での選択（ドリルダウン方式）

コンテナやグループが入れ子になっている場合、**段階的ドリルダウン方式**で選択階層を管理する。

#### 2.6.1 ドリルダウン操作

| 操作 | 動作 |
|------|------|
| クリック | 現在の `selectionContext` 階層で最上位の要素を選択 |
| ダブルクリック | 選択中のコンテナ/グループの中に「入る」。`selectionContext` を選択中要素のIDに更新 |
| ESC | `selectionContext` を1段上に戻す。最上位なら選択解除 |
| キャンバス空白クリック | `selectionContext` を null にリセット（最上位まで戻る） |
| Ctrl+クリック | ドリルダウンをスキップし、マウス位置の最深の要素を直接選択（`selectionContext` も自動更新） |

#### 2.6.2 ドリルダウンの具体例

```
[構造]
Root
├── Container A
│   ├── Rect 1
│   ├── Container B
│   │   ├── Rect 2
│   │   └── Rect 3
│   └── Rect 4
└── Rect 5

[操作フロー]
1. Rect 2 上をクリック → Container A が選択される（selectionContext = null）
2. ダブルクリック       → Container A 内に入る（selectionContext = "Container A"）
3. Rect 2 上をクリック → Container B が選択される（selectionContext = "Container A"）
4. ダブルクリック       → Container B 内に入る（selectionContext = "Container B"）
5. Rect 2 をクリック   → Rect 2 が選択される
6. ESC                 → Container A 内に戻る（selectionContext = "Container A"）
7. ESC                 → ルート階層に戻る（selectionContext = null）

[ショートカット]
手順1の代わりに Ctrl+クリック → Rect 2 を直接選択（selectionContext = "Container B"）
```

#### 2.6.3 ドリルダウン状態のUI表示

- 現在の `selectionContext` をパンくずリスト的に表示する（例: `Root > Container A > Container B`）
- ドリルダウン中のコンテナは、枠線を強調表示してコンテキスト階層を視覚的に示す
- パンくずリストのクリックで任意の階層にジャンプ可能

### 2.7 グループ内要素の選択

グループはコンテナと同じドリルダウン方式で操作する。

| 操作 | 動作 |
|------|------|
| グループをクリック | グループ全体を選択（移動・変形はグループ全体に適用） |
| グループをダブルクリック | グループ内に入り（`selectionContext` = グループID）、内部要素を個別に選択可能 |
| ESC | グループから出て、1段上の階層に戻る |

## 3. ドラッグ&ドロップ

### 3.1 要素の移動（基本）

選択中の要素をドラッグして移動する基本操作。

| 項目 | 仕様 |
|------|------|
| ドラッグ開始条件 | 選択中の要素上でマウスダウン → 5px 以上移動でドラッグ開始（閾値でクリックと区別） |
| 座標更新 | Konva.js の `dragmove` イベントで要素の `x`, `y` を更新 |
| スナップ連携 | グリッドスナップ・スマートガイドがONの場合、移動先の座標にスナップ処理を適用 |
| ロック要素 | `locked: true` の要素はドラッグ不可（Konva の `draggable: false` で制御） |
| 矢印キー移動 | 選択中の要素を矢印キーで 1px 移動、Shift+矢印キーで 10px 移動 |

#### 3.1.1 ドラッグ移動のフロー

```
1. 選択要素上でマウスダウン
2. マウス移動（5px以上）→ ドラッグ開始
   - beginTransaction() を呼び出し（Undo用トランザクション開始）
   - Konva.js のドラッグ処理が開始
3. ドラッグ中（dragmove イベント）
   - スナップ処理: グリッドスナップ / スマートガイドで座標を補正
   - スマートガイド: overlayLayer にガイドラインを表示
   - コンテナハイライト: ドラッグ先にコンテナがある場合、枠をハイライト
4. マウスアップ（dragend イベント）
   - 最終座標を ElementNode の x, y に反映
   - コンテナへのドロップ判定（セクション3.2）
   - commitTransaction() を呼び出し（Undo用トランザクション確定）
   - スマートガイド・ハイライトをクリア
```

### 3.2 コンテナ間の移動（親子関係の変更）

要素をドラッグしてコンテナ上にドロップした場合の親子関係変更。

#### 3.2.1 コンテナ自動認識

デフォルトでは、ドロップ先がコンテナであれば自動的に子要素として追加する。Alt キーを押しながらドロップすると、親子関係を設定せず上に重ねるだけとなる。

| 操作 | 動作 |
|------|------|
| コンテナ上にドロップ（Alt なし） | コンテナの子要素として追加 |
| コンテナ上にドロップ（Alt 押下） | 同じ階層に配置（親子関係を設定しない） |

#### 3.2.2 ドロップ判定のフロー

```
ドラッグ中:
  1. マウス位置のキャンバス座標を算出（ビューポート座標 → キャンバス座標変換）
  2. 現在の selectionContext 階層で、マウス位置に重なるコンテナを検索
     - ドラッグ中の要素自身と、ドラッグ中の要素の子孫は検索対象から除外
  3. コンテナが見つかった場合:
     - コンテナの枠をハイライト表示（青色の破線）
     - Flex/Grid コンテナの場合は挿入位置マーカーも表示

ドロップ時:
  Alt未押下の場合:
    1. ネスト深さチェック（推奨8階層/上限16階層）
       - 上限超過: 操作を拒否し、エラーメッセージを表示
       - 推奨超過: 操作を実行し、注意メッセージを表示
    2. 元の親の childrenIds から要素IDを除去
    3. 新しい親（コンテナ）の childrenIds に要素IDを追加
    4. 要素の parentId を新しい親のIDに更新
    5. 要素の x, y をコンテナのローカル座標に変換

  Alt押下の場合:
    → 座標の更新のみ。親子関係は変更しない
```

#### 3.2.3 循環参照の防止

コンテナをその子孫コンテナにドロップしようとした場合（循環参照になる操作）は、操作を拒否する。

```
判定: ドロップ先コンテナの祖先チェーンに、ドラッグ中の要素が含まれていないことを確認
```

### 3.3 Flex/Grid コンテナ内の順序変更

テーマ3 セクション3.7.1で定められた「`childrenIds` の順序変更で再配置」の原則に基づき、ドラッグ操作のUXを設計する。

#### 3.3.1 Flex コンテナ内のドラッグ

```
ドラッグ中:
  1. ドラッグ中の要素をゴースト表示（半透明）
  2. マウス位置に最も近い挿入位置を計算
     - Flex direction が row の場合: 隣接要素の中心のX座標で判定
     - Flex direction が column の場合: 隣接要素の中心のY座標で判定
  3. 挿入位置にマーカー（縦線 or 横線）を表示

ドロップ時:
  childrenIds 内の要素の位置を変更（配列内の移動）
  → レイアウトエンジンが自動的に再配置を計算
```

#### 3.3.2 Grid コンテナ内のドラッグ

```
ドラッグ中:
  1. ドラッグ中の要素をゴースト表示（半透明）
  2. マウス位置に対応するグリッドセルをハイライト
  3. ドロップ先セルに既に要素がある場合は、入れ替えを示すプレビュー

ドロップ時:
  childrenIds 内の要素の位置を変更（セル位置の入れ替え）
  → レイアウトエンジンが自動的に再配置を計算
```

#### 3.3.3 コンテナ外へのドラッグ

Flex/Grid コンテナの子要素をコンテナの外にドラッグした場合:

1. コンテナの `childrenIds` から要素IDを除去
2. 要素の `parentId` を null（ルート）またはドロップ先コンテナのIDに更新
3. 要素の `x`, `y` をドロップ先の座標系に変換
4. ドロップ先が別のコンテナの場合はセクション3.2の判定を適用

### 3.4 ツリービュー上の要素移動

レイヤーパネルのツリービュー上での要素移動は、コンテキストメニューおよびボタン操作で行う（ドラッグ&ドロップは使用しない）。

| 操作 | 説明 | データ操作 |
|------|------|-----------|
| ↑ 上に移動 | 同一親内で1つ上（`childrenIds` 内で1つ前）に移動 | `childrenIds` の順序変更 |
| ↓ 下に移動 | 同一親内で1つ下（`childrenIds` 内で1つ後）に移動 | `childrenIds` の順序変更 |
| → 中に入れる | 直下の兄弟コンテナ/グループの子要素として移動 | `parentId` と `childrenIds` の更新 |
| ← 外に出す | 親のコンテナ/グループから1つ上の階層に移動 | `parentId` と `childrenIds` の更新 |
| ⇈ 先頭に移動 | 同一親内で先頭に移動 | `childrenIds` の順序変更 |
| ⇊ 末尾に移動 | 同一親内で末尾に移動 | `childrenIds` の順序変更 |

ツリービューの要素移動操作は、キャンバス上のD&D（セクション3.1〜3.3）と同じデータ操作（`parentId`/`childrenIds` の更新）を行う。「中に入れる」操作時はネスト深さチェック（推奨8階層/上限16階層）を適用する。

なお、キャンバス上のドラッグ&ドロップによる要素の移動・親子関係変更は引き続き利用可能である。画面レイアウトの詳細は横断設計書07（07-screen-layout.md セクション7.2、セクション8.2）を参照。

### 3.5 複数要素のドラッグ

複数選択した要素をまとめてドラッグする場合の動作。

| 項目 | 仕様 |
|------|------|
| 移動 | 全選択要素が同じ差分（deltaX, deltaY）だけ移動 |
| スナップ | ドラッグの基準要素（最初にマウスダウンした要素）のみスナップ処理を適用 |
| コンテナへのドロップ | 全選択要素がまとめてコンテナの子要素に追加 |
| 異なる親の要素 | 異なるコンテナに属する要素を同時にドラッグ可能。ドロップ先の親子関係は一括で更新 |

### 3.6 ドラッグ中のフィードバック

ドラッグ操作中にユーザーに提供する視覚的フィードバック。

| フィードバック | 表示先 | 説明 |
|-------------|--------|------|
| ドラッグ中の要素 | elementLayer | 元の位置から実際に移動する。Konva.js の標準ドラッグ挙動 |
| コンテナハイライト | overlayLayer | ドロップ先候補のコンテナの枠を青色破線でハイライト |
| 挿入位置マーカー | overlayLayer | Flex/Grid コンテナ内の挿入位置を線/セルハイライトで表示 |
| スマートガイド | overlayLayer | 整列ガイドライン（テーマ3 セクション5.4の定義に従う） |
| ネスト深さ警告 | UI通知 | 推奨上限を超える場合に注意バッジを表示 |
| 自動パン | — | 初期実装では非対応。将来的にキャンバス端へのドラッグ時に自動パンを導入する候補 |

## 4. リサイズ・回転

### 4.1 Konva.Transformer の利用方針

要素のリサイズ・回転には Konva.js の Transformer を使用する。Transformer は overlayLayer に配置し、選択中の要素にアタッチする。

```typescript
// Transformer の基本設定
const transformerConfig = {
  // リサイズハンドル（8点: 四隅 + 四辺の中点）
  enabledAnchors: [
    'top-left', 'top-center', 'top-right',
    'middle-left', 'middle-right',
    'bottom-left', 'bottom-center', 'bottom-right',
  ],
  // 回転ハンドル
  rotateEnabled: true,
  // 回転ハンドルの位置（上辺中央の上方）
  rotateAnchorOffset: 30,
  // 最小サイズ制限
  boundBoxFunc: (oldBox, newBox) => {
    if (newBox.width < 10 || newBox.height < 10) {
      return oldBox;
    }
    return newBox;
  },
};
```

### 4.2 リサイズ操作

| 操作 | 動作 |
|------|------|
| ハンドルをドラッグ | 要素をリサイズ。`width`, `height` を更新 |
| Shift+ドラッグ | アスペクト比を固定してリサイズ |
| Alt+ドラッグ | 中心を基準にリサイズ（対角ハンドルが対称に移動） |
| Shift+Alt+ドラッグ | アスペクト比固定 + 中心基準リサイズ |

#### 4.2.1 最小サイズ制限

要素の最小サイズは 10px × 10px とする。これより小さくリサイズすることはできない。

#### 4.2.2 リサイズ時のデータ更新

```
transformend イベント発火時:
  1. Konva ノードの scaleX, scaleY を読み取り
  2. ElementNode の width, height を更新（width * scaleX, height * scaleY）
  3. Konva ノードの scaleX, scaleY を 1.0 にリセット
  4. x, y の位置補正（スケーリングによるオフセット）
```

> **設計根拠**: Konva.Transformer はリサイズ時にノードの `scale` を変更する。ElementNode は `width`/`height` で管理するため、`transformend` 時に `scale` → `width`/`height` に変換してリセットする。

### 4.3 回転操作

| 項目 | 仕様 |
|------|------|
| 回転ハンドル | 上辺中央の上方に表示 |
| 回転角度 | 自由回転（0〜360度） |
| Shift+回転 | 15度単位でスナップ |
| データ更新 | ElementNode の `rotation` を更新 |

### 4.4 コンテナリサイズ時の子要素の振る舞い

コンテナをリサイズした場合の内部の子要素への影響。

| レイアウト | 挙動 |
|-----------|------|
| Free | 子要素の位置・サイズは変わらない（コンテナのサイズのみ変更） |
| Flex | レイアウトエンジンが再計算。子要素の配置が更新される（`effectiveSize` が変更されるため） |
| Grid | レイアウトエンジンが再計算。セルサイズが変更され、子要素の配置が更新される |

> **Free レイアウトの補足**: コンテナのサイズ変更により、子要素がコンテナの範囲外にはみ出す場合がある。`overflow: 'clip'` の場合はクリッピングされるが、`overflow: 'visible'`（デフォルト）の場合はそのまま表示される。

### 4.5 複数要素の一括変形

複数選択時のリサイズ・回転は、Konva.Transformer の複数ノード対応機能を利用する。

| 項目 | 仕様 |
|------|------|
| 複数選択時のリサイズ | 選択要素全体のバウンディングボックスに対してリサイズ。各要素は相対的な位置・サイズ比を維持 |
| 複数選択時の回転 | バウンディングボックスの中心を基準に全要素が回転 |
| Transformer のアタッチ | 複数の Konva ノードを `transformer.nodes([...])` で指定 |

## 5. グループ化とコンテナの操作境界

### 5.1 グループ化/解除

#### 5.1.1 グループ化（Ctrl+G）

複数の要素を選択してグループ化する操作。

```
操作フロー:
  1. 複数要素を選択
  2. Ctrl+G を押下
  3. 新しい group タイプの ElementNode を作成
     - id: 新規生成（UUID）
     - type: 'group'
     - parentId: 選択要素の共通の親（全て同じ親であること）
     - childrenIds: 選択要素のID配列（元のZオーダーを維持）
     - x, y: 選択要素のバウンディングボックスの左上座標
     - width, height: バウンディングボックスのサイズ
  4. 選択要素の parentId をグループIDに更新
  5. 選択要素の x, y をグループのローカル座標に変換
  6. 元の親の childrenIds から選択要素を除去し、グループIDを挿入
     - 挿入位置: 選択要素の中で最も背面にあった位置
```

#### 5.1.2 グループ化の前提条件

| 条件 | 説明 |
|------|------|
| 2つ以上の要素を選択 | 1つの要素ではグループ化不可 |
| 同一親の要素のみ | 異なるコンテナに属する要素のグループ化は不可 |

#### 5.1.3 グループ解除（Ctrl+Shift+G）

```
操作フロー:
  1. グループを選択
  2. Ctrl+Shift+G を押下
  3. グループの childrenIds の各要素について:
     - parentId をグループの parentId に更新
     - x, y をグループの親のローカル座標に変換（グループの座標を加算）
  4. グループの親の childrenIds にグループの子要素を挿入（グループの位置に）
  5. グループの ElementNode を削除
  6. 元のグループの子要素を全て選択状態にする
```

### 5.2 エッジケースの設計

#### 5.2.1 エッジケース①: グループ内の要素をコンテナにドロップ

グループ内の要素をドリルダウンで選択し、コンテナにドロップした場合の動作。

**動作: グループから離脱してコンテナの子になる**

```
[操作前]
Root
├── Group A
│   ├── Rect 1
│   ├── Rect 2    ← ドラッグ
│   └── Rect 3
└── Container B

[操作: Rect 2 を Container B にドロップ]

[操作後]
Root
├── Group A
│   ├── Rect 1
│   └── Rect 3
└── Container B
    └── Rect 2

[残り1要素の場合]
Group A の子要素が Rect 3 のみとなった場合:
→ グループを自動解除（Rect 3 が Root の直接の子になる）
```

- ユーザーが明示的に要素をコンテナにドロップしている以上、その意図を尊重する
- 単一要素のグループは無意味なため、残り1要素の場合は自動解除する

#### 5.2.2 エッジケース②: グループとコンテナの相互変換

**双方向の変換機能を提供する。**

| 変換方向 | 操作方法 | 内部処理 |
|---------|---------|---------|
| グループ → コンテナ | 右クリックメニュー「コンテナに変換」 | `type` を `'container'` に変更。`layout`（FreeLayout）、`padding`（全方向0）、`overflow`（'visible'）をデフォルト値で追加。`props` を ContainerProps に変更 |
| コンテナ → グループ | 右クリックメニュー「グループに変換」 | `type` を `'group'` に変更。`layout`、`padding`、`overflow` を除去。`props` を除去 |

**コンテナ → グループ変換時の注意事項:**

- Flex/Grid レイアウトが設定されている場合、変換時にレイアウト情報が消失する
- 変換前に**確認ダイアログ**を表示: 「レイアウト設定が失われます。変換しますか？」
- 変換後、子要素の `x`, `y` にはレイアウト計算で使用していた `computedX`, `computedY` を書き込み、Free配置として位置を維持する

#### 5.2.3 エッジケース③: コンテナ内の要素をグループ化

コンテナ内で複数要素を選択してグループ化した場合の動作。

**動作: コンテナの直接の子としてグループを作成**

```
[操作前]
Root
└── Container A (Flex)
    ├── Rect 1
    ├── Rect 2    ← 選択
    ├── Rect 3    ← 選択
    └── Rect 4

[操作: Rect 2 と Rect 3 をグループ化]

[操作後]
Root
└── Container A (Flex)
    ├── Rect 1
    ├── Group B          ← Container A の子として作成
    │   ├── Rect 2
    │   └── Rect 3
    └── Rect 4
```

- グループは親コンテナの子要素として自然に存在する
- Flex/Grid レイアウトのコンテナ内でグループ化した場合、グループが1つのレイアウトアイテムとして配置される
- グループ解除時は、子要素が親コンテナの直接の子に戻る

### 5.3 グループとコンテナの操作上の区別（まとめ表）

| 操作 | グループ | コンテナ |
|------|---------|---------|
| 選択 | ドリルダウン方式（クリック→グループ全体、ダブルクリック→内部） | ドリルダウン方式（同左） |
| 移動 | グループ全体が移動 | コンテナ全体が移動（子要素も追従） |
| リサイズ | バウンディングボックスのリサイズ。子要素は相対的にスケーリング（※1） | コンテナサイズの変更。Free: 子要素変化なし、Flex/Grid: レイアウト再計算 |
| レイアウト | なし（子要素は自由配置） | Free / Flex / Grid |
| padding | なし | あり |
| overflow | なし（子要素は常に表示） | visible / clip |
| D&D受け入れ | グループ内に外部から要素をドロップ不可 | コンテナ内に要素をドロップ可（子要素として追加） |
| 相互変換 | 右クリック「コンテナに変換」 | 右クリック「グループに変換」（レイアウト消失警告あり） |
| 子要素のグループ化 | — | コンテナの子としてグループを作成 |

> **※1 グループリサイズ時の子要素スケーリング**: グループのリサイズ時は、子要素の `x`, `y`, `width`, `height` をリサイズ比率に応じて再計算する。具体的には、グループの新旧サイズの比率（`scaleX = newWidth / oldWidth`, `scaleY = newHeight / oldHeight`）を算出し、各子要素の座標・サイズにこの比率を乗じる。Konva の `scale` プロパティは使用せず、ElementNode の値を直接更新する（scale は常に 1.0 を維持する方針との整合性を保つため）。

## 6. Undo/Redo

### 6.1 コマンドパターン（プロパティ差分方式）

Undo/Redo は**プロパティ差分（Patch）方式**で実装する。操作ごとにコマンドクラスを定義するのではなく、変更前後のプロパティ差分を汎用的に記録する。

#### 6.1.1 データ構造

```typescript
/** Undo/Redo の1エントリ（1つの操作に対応） */
interface UndoEntry {
  /** 操作の説明（UI表示用。例: "要素を移動", "3個の要素を削除"） */
  description: string;

  /** 操作のタイムスタンプ */
  timestamp: number;

  /** 変更差分の配列（1操作で複数の変更が発生する場合） */
  patches: Patch[];
}

/** 個別の変更差分 */
interface Patch {
  /** 変更の対象種別 */
  target: 'element' | 'page';

  /** 変更の種別 */
  type: 'update' | 'add' | 'remove';

  /** 対象要素ID（target が 'element' の場合） */
  elementId?: string;

  /** 変更前の値（Undo 用） */
  before: Record<string, unknown> | null;

  /** 変更後の値（Redo 用） */
  after: Record<string, unknown> | null;
}
```

#### 6.1.2 Patch の記録例

**要素の移動（x, y の変更）:**

```typescript
{
  description: "要素を移動",
  timestamp: Date.now(),
  patches: [
    {
      target: 'element',
      type: 'update',
      elementId: 'rect-1',
      before: { x: 100, y: 200 },
      after: { x: 150, y: 250 },
    },
  ],
}
```

**要素の追加:**

```typescript
{
  description: "矩形を追加",
  timestamp: Date.now(),
  patches: [
    {
      target: 'element',
      type: 'add',
      elementId: 'rect-new',
      before: null,
      after: { /* ElementNode 全体 */ },
    },
    {
      target: 'page',
      type: 'update',
      before: { rootElementIds: ['rect-1', 'rect-2'] },
      after: { rootElementIds: ['rect-1', 'rect-2', 'rect-new'] },
    },
  ],
}
```

**親子関係の変更（コンテナへのドロップ）:**

```typescript
{
  description: "要素をコンテナに移動",
  timestamp: Date.now(),
  patches: [
    {
      target: 'element',
      type: 'update',
      elementId: 'rect-1',
      before: { parentId: null, x: 100, y: 200 },
      after: { parentId: 'container-a', x: 50, y: 80 },
    },
    {
      target: 'page',
      type: 'update',
      before: { rootElementIds: ['rect-1', 'container-a'] },
      after: { rootElementIds: ['container-a'] },
    },
    {
      target: 'element',
      type: 'update',
      elementId: 'container-a',
      before: { childrenIds: [] },
      after: { childrenIds: ['rect-1'] },
    },
  ],
}
```

#### 6.1.3 Undo/Redo の実行

```
Undo:
  1. undoStack の末尾から UndoEntry を取り出す
  2. patches を逆順に走査し、各 Patch の before の値をストアに適用
  3. 取り出した UndoEntry を redoStack に積む

Redo:
  1. redoStack の末尾から UndoEntry を取り出す
  2. patches を順方向に走査し、各 Patch の after の値をストアに適用
  3. 取り出した UndoEntry を undoStack に積む

新規操作が記録された場合:
  → redoStack をクリアする（分岐履歴は破棄）
```

#### 6.1.4 設計根拠

| 観点 | Patch 方式（採用） | コマンドオブジェクト方式 | スナップショット方式 |
|------|-------------------|--------------------|--------------------|
| 実装コスト | 低い（汎用的な1つの仕組み） | 高い（操作ごとにクラス定義） | 低い |
| メモリ効率 | 良い（差分のみ保持） | 良い | 悪い（状態全体を保持） |
| フラットマップとの相性 | ◎（キーの差分記録が自然） | ○ | ○ |
| デバッグ | 差分を見れば変更内容が明確 | コマンドごとの理解が必要 | 比較が容易 |

### 6.2 スコープ（ページ単位）

Undo/Redo の履歴は**ページ単位**で管理する。

```typescript
interface UndoManager {
  /** ページIDごとの Undo スタック */
  undoStacks: Record<string, UndoEntry[]>;

  /** ページIDごとの Redo スタック */
  redoStacks: Record<string, UndoEntry[]>;

  /** 現在アクティブなページID */
  activePageId: string;
}
```

| 操作 | 挙動 |
|------|------|
| Ctrl+Z | アクティブページの undoStack から最新エントリを取り出して Undo |
| Ctrl+Shift+Z / Ctrl+Y | アクティブページの redoStack から最新エントリを取り出して Redo |
| ページ切り替え | `activePageId` を更新。各ページの履歴は独立して保持される |
| ページ削除 | 該当ページの undoStack / redoStack を削除 |

#### 6.2.1 ページ単位の設計根拠

1. ページAで作業 → ページBに移動 → ページAに戻る → Undo → **ページAの操作が取り消される**（直感的）
2. プロジェクト全体方式だと、上記の流れでページBの操作がUndoされ、混乱する
3. Figma 等の主要デザインツールと同じ方式
4. 非アクティブページの詳細な履歴を保持し続ける必要がなく、メモリ効率が良い

### 6.3 デフォルト履歴数と設定

| 項目 | 値 |
|------|-----|
| デフォルト履歴数 | 50 |
| 設定可能範囲 | 10 〜 200 |
| 保存先 | ProjectSettings（プロジェクト単位の設定） |

履歴数が上限に達した場合、最も古い UndoEntry から削除する（FIFO）。

### 6.4 履歴管理の仕組み

#### 6.4.1 ストアの構造

```typescript
// Pinia ストア（Undo/Redo管理）
const useUndoStore = defineStore('undo', () => {
  const undoStacks = ref<Record<string, UndoEntry[]>>({});
  const redoStacks = ref<Record<string, UndoEntry[]>>({});

  const maxHistory = ref<number>(50);

  // トランザクション管理
  const pendingTransaction = ref<UndoEntry | null>(null);

  function undo() { /* ... */ }
  function redo() { /* ... */ }

  function beginTransaction(description: string) { /* ... */ }
  function commitTransaction() { /* ... */ }
  function rollbackTransaction() { /* ... */ }

  const canUndo = computed(() => /* ... */);
  const canRedo = computed(() => /* ... */);
});
```

#### 6.4.2 isDirty フラグとの連携

テーマ2で定義された `isDirty` フラグは、Undo/Redo 操作でも更新される。

| 操作 | isDirty の変化 |
|------|---------------|
| 要素の編集 | true に設定 |
| Undo | 全Undoで保存時点に戻った場合のみ false |
| Redo | true に設定 |
| 保存 | false にリセット |

### 6.5 連続操作の結合（トランザクション方式）

ドラッグ移動やリサイズなど、連続的に発生する変更を1つの Undo エントリにまとめるためにトランザクション機構を使用する。

#### 6.5.1 トランザクションのAPI

```typescript
// トランザクション開始: 以降の変更を一時バッファに蓄積
beginTransaction(description: string): void

// トランザクション確定: バッファの内容を1つの UndoEntry にまとめて記録
commitTransaction(): void

// トランザクション破棄: バッファの内容を破棄し、変更をロールバック
rollbackTransaction(): void
```

#### 6.5.2 トランザクションの適用対象

| 操作 | トランザクション境界 | 説明 |
|------|-------------------|------|
| ドラッグ移動 | dragstart 〜 dragend | ドラッグ中の全座標変更を1エントリにまとめる |
| リサイズ | transformstart 〜 transformend | リサイズ中の全サイズ変更を1エントリにまとめる |
| 回転 | transformstart 〜 transformend | 回転中の全角度変更を1エントリにまとめる |
| テキスト編集 | 編集モード開始 〜 確定 | テキスト変更全体を1エントリにまとめる |
| プロパティスライダー | pointerdown 〜 pointerup | スライダー操作中の連続変更を1エントリにまとめる |

#### 6.5.3 トランザクションの最適化

トランザクション内で同一要素の同一プロパティが複数回変更された場合、最初の before と最後の after のみを保持する。

```
例: ドラッグ中に x が 100 → 120 → 150 → 180 と変化した場合

記録される Patch:
  before: { x: 100 }   （最初の値）
  after:  { x: 180 }   （最後の値）
```

### 6.6 ページ間コピー&ペースト時の Undo 挙動

| 操作 | Undo の挙動 |
|------|------------|
| ページAでコピー → ページBでペースト | ページBの undoStack にペースト操作が記録される。ページBでUndoするとペーストが取り消される。ページAには影響なし |
| ページAでカット → ページBでペースト | ページAの undoStack にカット操作が記録。ページBの undoStack にペースト操作が記録。各ページで独立してUndoできる |

## 7. その他の操作

### 7.1 コピー&ペースト / カット / 複製

#### 7.1.1 コピー（Ctrl+C）

選択中の要素のデータ（ElementNode のディープコピー）を内部クリップボードに保存する。

| 項目 | 仕様 |
|------|------|
| コピー対象 | 選択中の全要素（子孫要素を含む完全なサブツリー） |
| クリップボード | アプリケーション内部のクリップボード（ブラウザのクリップボードAPIは使用しない） |
| コピー元の状態 | 変化なし |

> **設計根拠**: ブラウザのクリップボードAPIを使用しない理由は、ElementNodeの構造データは外部アプリとの互換性がなく、ブラウザの権限管理の複雑さを避けるため。

#### 7.1.2 ペースト（Ctrl+V）

クリップボードの内容をアクティブページに貼り付ける。

```
ペースト処理:
  1. クリップボードから ElementNode のデータを取得
  2. 全要素に新しい ID を生成（UUID）
  3. parentId / childrenIds の参照を新しい ID に更新
  4. ペースト位置:
     - 同一ページへのペースト: 元の位置から (20, 20) オフセット
     - 異なるページへのペースト: キャンバスの中央付近
  5. selectionContext のコンテナ内にペースト（selectionContext が null ならルートに追加）
  6. ネスト深さチェック（上限超過時は拒否）
  7. Undo エントリを記録
  8. ペーストした要素を選択状態にする
```

#### 7.1.3 カット（Ctrl+X）

コピー + 元の要素の削除を1つの操作として実行する。

```
カット処理:
  1. 選択要素をクリップボードにコピー
  2. 選択要素を削除（親の childrenIds からの除去を含む）
  3. Undo エントリを記録（削除操作として）
```

#### 7.1.4 複製（Ctrl+D）

選択中の要素のコピーを同じ位置（(20, 20) オフセット）に即座に作成する。コピー→ペーストのショートカット。

### 7.2 Zオーダー変更

`childrenIds` 配列内の要素位置を変更することでZオーダーを操作する。

| 操作 | ショートカット | 動作 |
|------|-------------|------|
| 最前面へ | Ctrl+Shift+] | 要素を `childrenIds` の末尾に移動 |
| 1つ前面へ | Ctrl+] | 要素を `childrenIds` 内で1つ後ろに移動 |
| 1つ背面へ | Ctrl+[ | 要素を `childrenIds` 内で1つ前に移動 |
| 最背面へ | Ctrl+Shift+[ | 要素を `childrenIds` の先頭に移動 |

操作対象:
- ルート要素の場合: ページの `rootElementIds` 配列内で位置を変更
- コンテナ/グループの子の場合: 親の `childrenIds` 配列内で位置を変更

### 7.3 整列・均等配置

複数要素を選択した場合に使用可能な整列・均等配置操作。

#### 7.3.1 整列操作

| 操作 | 動作 |
|------|------|
| 左揃え | 全選択要素の左端を、最も左にある要素の左端に揃える |
| 右揃え | 全選択要素の右端を、最も右にある要素の右端に揃える |
| 上揃え | 全選択要素の上端を、最も上にある要素の上端に揃える |
| 下揃え | 全選択要素の下端を、最も下にある要素の下端に揃える |
| 水平中央揃え | 全選択要素の水平中心を、選択範囲の水平中央に揃える |
| 垂直中央揃え | 全選択要素の垂直中心を、選択範囲の垂直中央に揃える |

#### 7.3.2 均等配置

| 操作 | 動作 |
|------|------|
| 水平に均等配置 | 選択要素間の水平間隔を均等にする |
| 垂直に均等配置 | 選択要素間の垂直間隔を均等にする |

#### 7.3.3 操作条件

| 操作 | 必要な選択数 |
|------|------------|
| 整列 | 2つ以上 |
| 均等配置 | 3つ以上 |

整列・均等配置操作は Free レイアウト（またはルート要素）の場合のみ有効。Flex/Grid コンテナ内の要素にはレイアウトエンジンが配置を制御するため、手動の整列は適用しない。

### 7.4 テキスト編集

テーマ3 セクション2.5.3で定義されたテキスト編集フローに基づく操作設計。

| 操作 | 動作 |
|------|------|
| テキスト要素をダブルクリック | テキスト編集モードに入る |
| 編集中に ESC | 編集を確定して通常モードに戻る |
| 編集中に外部クリック | 編集を確定して通常モードに戻る |
| 編集中に Ctrl+A | テキスト全選択（DOM の textarea 内の操作） |

#### 7.4.1 テキスト編集とドリルダウンの関係

テキスト要素がコンテナ内にある場合:

```
1. コンテナをクリック → コンテナ選択（selectionContext = null）
2. ダブルクリック → コンテナ内に入る（selectionContext = container-id）
3. テキスト要素をクリック → テキスト要素選択
4. ダブルクリック → テキスト編集モード開始
```

テキスト要素がルート階層にある場合:

```
1. テキスト要素をクリック → テキスト要素選択
2. ダブルクリック → テキスト編集モード開始
```

#### 7.4.2 テキスト編集の Undo

テキスト編集はトランザクションで括る。編集モード開始時に `beginTransaction("テキストを編集")` を呼び出し、確定時に `commitTransaction()` を呼び出す。ESC キーで編集を破棄する場合は `rollbackTransaction()` を呼び出す。

### 7.5 ページ間リンク設定

要素にリンク先ページを設定する操作。

| 項目 | 仕様 |
|------|------|
| 設定方法 | プロパティパネルからリンク先ページを選択 |
| リンク先 | 同一プロジェクト内の別ページ |
| データ | ElementNode の props にリンク先ページIDを保持 |
| 表示 | リンクが設定された要素にはリンクアイコンをオーバーレイ表示 |
| プレビュー操作 | プレビューモードでリンク要素をクリックするとリンク先ページに遷移 |

### 7.6 要素のロック/アンロック

| 操作 | ショートカット | 動作 |
|------|-------------|------|
| ロック | Ctrl+L | 選択要素の `locked` を true に設定 |
| アンロック | Ctrl+L | 選択要素の `locked` を false に設定（トグル） |

ロック中の制限:

| 操作 | ロック時の挙動 |
|------|-------------|
| 選択 | 可能（プロパティパネルで情報表示） |
| ドラッグ | 不可（`draggable: false`） |
| リサイズ | 不可（Transformer 非表示） |
| 回転 | 不可（Transformer 非表示） |
| プロパティ編集 | 不可 |
| 削除 | 可能（ロックは削除を防止しない） |
| コピー | 可能 |

### 7.7 要素の削除

Delete / Backspace キーで選択中の要素を削除する。

```
削除処理フロー:
  1. 選択中の全要素について:
     a. 要素がコンテナ/グループの場合、子孫要素を再帰的に収集
     b. 親の childrenIds（またはページの rootElementIds）から要素IDを除去
  2. 収集した全要素（選択要素 + 子孫要素）を elements フラットマップから削除
  3. Undo エントリを記録（削除した全要素の before 状態を保存）
  4. selectedIds をクリア
```

| 項目 | 仕様 |
|------|------|
| 子孫の再帰削除 | コンテナ/グループを削除すると、その子孫要素も全て削除される |
| Undo | 削除操作のUndoで、全要素が元の位置・親子関係で復元される |
| ロック要素 | ロック中でも削除可能 |
| 非表示要素 | ツリービューから選択すれば削除可能 |

## 8. キーボードショートカット一覧

### 8.1 設計方針

| 方針 | 説明 |
|------|------|
| 業界標準に準拠 | Figma, Sketch, Adobe XD の標準ショートカットに可能な限り揃える |
| Chrome 競合回避 | Chromium ブラウザのデフォルトショートカット（Ctrl+T, Ctrl+W 等）と競合する場合は `e.preventDefault()` で奪取 |
| PWA 対応 | PWA インストール時はブラウザショートカットとの競合が発生しないため、全ショートカットが利用可能 |
| Ctrl 系のみ | 対応ブラウザが Chromium 系限定のため、Cmd 系（macOS）の考慮は不要 |
