## 1. アプリケーション形態

### 1.1 決定事項

**SPA（Single Page Application）+ PWA（Progressive Web App）** として開発する。

- ブラウザ上で動作するWebアプリケーションとして実装する
- PWA技術（Service Worker、Web App Manifest）を用いてオフライン動作を実現する
- PWAとしてデバイスにインストールすることも可能とする

### 1.2 決定理由

| 判断軸 | SPA + PWA | Electron | Tauri |
|--------|-----------|----------|-------|
| オフライン動作 | Service Worker で実現可能 | 標準で対応 | 標準で対応 |
| ファイルシステムアクセス | File System Access API（Chromium系で対応） | Node.js fsモジュールで自由 | Rust側でフル制御 |
| 配布・インストール | URL共有で即利用可能 / PWAインストール可能 | インストーラー配布が必要 | インストーラー配布が必要 |
| バンドルサイズ | アプリ本体のみ（軽量） | Chromium同梱（150MB〜） | OS標準WebView利用（10MB程度） |
| 開発コスト | Web技術のみで完結 | Web技術 + Node.jsブリッジ | Web技術 + Rustブリッジ |
| アップデート | デプロイ即反映 | 更新配布が必要 | 更新配布が必要 |

以下の理由から SPA + PWA を選定した。

1. **開発コストの低さ**: Web技術のみで完結し、ブリッジ層の開発が不要
2. **個人利用**: 配布の容易さやマルチプラットフォーム対応の優先度が低く、ローカルでの開発サーバー起動でも十分に運用可能
3. **オフライン動作**: Service Worker + Cache API により、要件定義書（セクション2.2）のオフライン動作必須要件を満たせる
4. **ファイル操作**: Chromium系ブラウザに限定することで、File System Access API による自然なファイル保存/読み込みが可能

## 2. 対応ブラウザ・動作環境

### 2.1 決定事項

**Chromium系ブラウザのみ**を対応対象とする。

| ブラウザ | 対応 |
|----------|------|
| Google Chrome | 対応 |
| Microsoft Edge（Chromium版） | 対応 |
| Firefox | 非対応 |
| Safari | 非対応 |
| その他Chromium系（Brave, Vivaldi等） | 動作する可能性があるが、サポート対象外 |

### 2.2 最低対応バージョン方針

- 開発時点での最新安定版から2バージョン前までを対応とする
- 具体的なバージョン番号は開発着手時に確定する

### 2.3 決定理由

1. **File System Access API の活用**: 独自ファイル形式の保存/読み込み（要件定義書セクション4.3）に、ネイティブのファイルダイアログ（開く/保存）を使用するため。このAPIはChromium系ブラウザのみ対応している
2. **個人利用**: 対象ユーザーが開発者本人のみ（要件定義書セクション1.2）であり、普段使用するブラウザがChromeであるため、マルチブラウザ対応の必要性が低い
3. **先進的Web APIの活用**: Chromium系に限定することで、Origin Private File System（OPFS）等の先進的なAPIも必要に応じて活用できる

## 3. キャンバス描画技術

### 3.1 決定事項

**Canvas 2D をメインの描画技術とし、DOMオーバーレイを併用する**。

Canvas 2Dライブラリとして **Konva.js**（+ Vue.js連携用の **vue-konva**）を採用する。

#### 描画方式の構成

| レイヤー | 技術 | 用途 |
|----------|------|------|
| メイン描画 | Canvas 2D（Konva.js） | 図形要素、UIコンポーネント、コンテナ等すべての要素の描画 |
| UI補助 | Canvas 2D（Konva.js） | グリッド、ガイドライン、選択範囲、変形ハンドル等 |
| DOMオーバーレイ | HTML DOM | テキスト編集時の入力フィールド、ツールチップ等 |

#### Konva.js の役割

- Canvas上のノード（図形）をツリー構造で管理する
- ヒットテスト（要素のクリック判定）を提供する
- ドラッグ＆ドロップのイベント処理を提供する
- 要素の変形（移動、リサイズ、回転）のインタラクションを提供する

#### Konva.js Group と本ツールの概念の関係

Konva.jsの「Group」は、本ツールにおける「**グループ**」（表示上の便宜的まとまり）と「**コンテナ**」（構造的親子関係）の**両方の描画上の基盤**として使用する。Konva.js Groupはあくまで描画レベルでのノードのまとまりを管理する仕組みであり、「グループ」と「コンテナ」の概念的な区別（レイアウト機能の有無、パディング/マージンの有無等）はドメイン層のデータモデルで管理する。

※ この区別の詳細設計はテーマ3（キャンバス・レンダリング設計）およびテーマ4（操作・インタラクション設計）で行う。

#### 自前実装が必要な領域

- **レイアウトエンジン**: Flexbox的/Grid的な自動レイアウトの計算ロジック（テーマ3で設計）
- **エクスポート変換**: 内部データモデルからHTML/CSS、各種XAML、PNG、SVGへの変換（テーマ5で設計）

### 3.2 選定理由・比較

| 観点 | Canvas 2D（Konva.js） | SVG | HTML/CSS DOM |
|------|----------------------|-----|-------------|
| パフォーマンス（要素数） | 良好（数千要素対応可能） | 数百要素で低下の可能性 | 数百要素で低下の可能性 |
| 親子構造の管理 | Konva.jsのGroupでツリー構造を管理（※1） | SVG要素の入れ子で自然 | DOM自体が親子構造 |
| Flexbox/Grid的レイアウト | 自前実装が必要 | 自前実装が必要 | CSSネイティブで利用可能 |
| ヒットテスト | Konva.jsが提供 | DOMイベントで自然 | DOMイベントで自然 |
| テキスト描画 | DOMオーバーレイで補完 | 高品質 | 高品質 |
| SVGエクスポートとの親和性 | 変換が必要 | ほぼそのまま出力可能 | 変換が必要 |
| ズーム・パン | 変換行列で対応 | viewBox変換で容易 | CSS transformで対応 |

以下の理由からCanvas 2D（Konva.js）を選定した。

1. **パフォーマンス**: 要素数の増加に対して最もスケーラブル。個人利用のワイヤーフレームツールとして実用的な要素数（数百〜千要素）でスムーズに動作することが求められる
2. **レイアウトエンジンの独立性**: 本ツールではエクスポート機能（HTML/CSS、各種XAML）のために、描画技術に依存しない抽象的なレイアウトモデルと計算エンジンを持つ必要がある。描画技術自体にレイアウト機能を委ねるメリットが薄いため、描画パフォーマンスで優れるCanvas 2Dが合理的
3. **Konva.jsの親和性**: ノードのツリー構造管理がコンテナの親子関係と構造的に一致する。vue-konvaによるVue.jsとの自然な連携も利点

※1: Konva.jsのGroupは描画レベルのノード管理であり、本ツールの「グループ」「コンテナ」両方の基盤として使用する。概念の区別はドメイン層で管理する（セクション3.1参照）。

## 4. 技術スタック

### 4.1 技術スタック一覧

| カテゴリ | 技術 | 備考 |
|----------|------|------|
| 言語 | TypeScript | 型安全性、エディタサポートの充実 |
| フレームワーク | Vue.js (Vue 3, Composition API) | UIパネル・ツールバー等の実装に使用 |
| Canvasライブラリ | Konva.js + vue-konva | キャンバス描画・操作の基盤 |
| 状態管理 | Pinia | Vue.js公式推奨の状態管理ライブラリ |
| ビルドツール | Vite | 高速なHMR、Vue.jsとの標準的な組み合わせ |
| ユニットテスト | Vitest | Viteベース、TypeScriptネイティブ対応 |
| E2Eテスト | Playwright | Canvas操作を含むE2Eテスト |
| CSS方針 | Vuetifyスタイルシステム + Vue SFC scoped CSS | 横断設計書07（07-screen-layout.md）で決定。ユーティリティCSS（UnoCSS / Tailwind CSS等）は不使用 |
| UIコンポーネントライブラリ | Vuetify | 横断設計書07（07-screen-layout.md）で決定。詳細なコンポーネントマッピングは07-screen-layout.md セクション10.3を参照 |

### 4.2 各技術の選定理由

#### TypeScript

- 型安全性により、要素のプロパティ定義やレイアウトモデルの型定義をコンパイル時に検証できる
- エディタの補完・リファクタリング支援が充実しており、個人開発の生産性向上に寄与する

#### Vue.js

- UIパネル（プロパティパネル、ツリービュー、ページ一覧等）のコンポーネント実装に使用する
- Composition APIにより、ロジックの再利用やテスタビリティが高い
- vue-konvaによるKonva.jsとの自然な連携が可能

#### Konva.js + vue-konva

- Canvas上にノードのツリー構造を構築でき、本ツールのコンテナ親子関係の管理と構造的に一致する
- ヒットテスト、ドラッグ&ドロップ、イベント処理、変形操作（Transformer）を標準提供する
- vue-konvaにより、VueコンポーネントとしてKonvaノードを宣言的に記述可能

#### Pinia

- Vue.js公式推奨の状態管理ライブラリ
- TypeScriptとの相性が良く、型推論が効く
- アプリケーション状態（現在のプロジェクト、選択中の要素、ツール状態等）を管理する

#### Vite

- Vue.jsプロジェクトの標準的なビルドツール
- 高速なHot Module Replacement（HMR）による開発体験の向上
- PWA対応プラグイン（vite-plugin-pwa）が利用可能

#### Vitest + Playwright

- Vitest: Viteベースのテストフレームワーク。ユニットテスト・コンポーネントテストに使用
- Playwright: ブラウザ自動化テストフレームワーク。Canvas操作を含むE2Eテストに使用

## 5. アプリケーションアーキテクチャ概要

### 5.1 レイヤー構成

本アプリケーションは以下のレイヤー構成を採用する。

```
┌─────────────────────────────────────────────┐
│              UI層 (Presentation)             │
│  Vue.js コンポーネント + vue-konva           │
│  ツールバー、パネル、キャンバス表示           │
├─────────────────────────────────────────────┤
│          アプリケーション層 (Application)     │
│  Pinia ストア、ユースケース（操作コマンド）   │
│  Undo/Redo管理、選択管理、ツール状態管理     │
├─────────────────────────────────────────────┤
│            ドメイン層 (Domain)               │
│  要素モデル、コンテナ構造、レイアウトエンジン │
│  プロジェクト・ページモデル                   │
│  エクスポート変換ロジック（変換ルール）       │
├─────────────────────────────────────────────┤
│         インフラ層 (Infrastructure)          │
│  ファイル保存/読み込み（File System Access    │
│  API）、IndexedDB、Service Worker            │
│  エクスポート出力（ファイル書き出し）         │
└─────────────────────────────────────────────┘
```

各レイヤーの責務は以下の通り。

| レイヤー | 責務 | 主要な構成要素 |
|----------|------|---------------|
| UI層 | ユーザーインタラクションの処理と画面表示 | Vueコンポーネント、vue-konvaノード、DOMオーバーレイ |
| アプリケーション層 | ユースケースの実行とアプリケーション状態の管理 | Piniaストア、操作コマンド（Undo/Redo対応）、選択管理 |
| ドメイン層 | ビジネスロジック、データモデルの定義、エクスポート変換ロジック | 要素型定義、コンテナ構造、レイアウト計算、バリデーション、各ターゲットへの変換ルール（テーマ5で詳細設計） |
| インフラ層 | 外部リソースとの入出力 | ファイル操作、ローカルストレージ、エクスポートファイル出力 |

### 5.2 オフライン動作方式

PWAの技術を用いてオフライン動作を実現する。

| 技術 | 用途 |
|------|------|
| **Service Worker** | アプリケーション自体のキャッシュ。オフライン時でもアプリを起動・使用可能にする |
| **Cache API** | アプリのHTML/CSS/JS等の静的アセットをキャッシュする |
| **Web App Manifest** | PWAとしてのインストール、アイコン、起動画面等を定義する |
| **File System Access API** | プロジェクトファイルの保存/読み込み。ローカルファイルシステムに直接アクセスする |
| **IndexedDB** | アプリケーション内部データの永続化（自動保存、設定情報等）。テーマ2で詳細設計 |

※ Service Workerの実装方式（vite-plugin-pwa（Workbox）による自動生成 / 手書き実装等）は、テーマ2（データ設計・永続化）で決定する。

#### オフライン時の動作範囲

- アプリケーションの起動と全基本機能の使用が可能
- プロジェクトの作成、編集、保存、読み込みが可能
- エクスポート（画像、コード）が可能
- オンライン接続が不要な完全ローカル動作
