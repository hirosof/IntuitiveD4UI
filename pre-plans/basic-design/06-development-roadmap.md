## 1. 概要

### 1.1 本設計書の範囲

本設計書は、Intuitive D4UI の開発を段階的に進めるためのフェーズ分けを定義する。要件定義書（セクション5）のフェーズ分け方針案を出発点に、テーマ1〜5および横断07の設計結果を踏まえて、各フェーズの詳細スコープ・依存関係・リスク対応を確定する。

本設計書で解決する検討事項:
- **検討事項No.8**: フェーズ分けの確定
- **検討事項No.B**: 遷移図ダイアグラムのフェーズ配置

### 1.2 各テーマ設計書からの入力

| テーマ | 本設計書への入力 |
|--------|----------------|
| テーマ1（アプリケーション基盤） | 技術スタック確定に基づくフェーズ別実現性評価 |
| テーマ3（キャンバス・レンダリング設計） | flex-grow/Grid伸縮のフェーズ2先送り、パフォーマンス最適化ロードマップ |
| テーマ4（操作・インタラクション設計） | ショートカットカスタマイズのフェーズ2先送り |
| テーマ5（エクスポート・変換設計） | フェーズ4/5のスコープ定義、PoC計画（3件） |
| 横断07（画面レイアウト設計） | Vuetify導入手順、自前拡張コンポーネントの実装範囲 |

## 2. フェーズ概要

### 2.1 フェーズ一覧

| フェーズ | 名称 | 概要 |
|---------|------|------|
| **フェーズ1** | 基盤構築・基本機能 | 技術基盤の構築と基本図形によるワイヤーフレーム作成 |
| **フェーズ2** | UIコンポーネント・拡張機能 | プリセットUI部品の追加とレイアウト・編集機能の拡充 |
| **フェーズ3** | プロジェクト管理・テンプレート | 複数ページ管理、ページ間リンク、遷移プレビュー、テンプレート |
| PoC-1 | HTML/CSS変換検証 | フェーズ4開始前の実現性検証 |

| **フェーズ5** | XAMLエクスポート | 4つのXAMLフレームワーク向けコードエクスポート |

### 2.2 フェーズ間の依存関係

```
フェーズ1（基盤 + 基本機能）
   │  全フェーズの土台。技術基盤・データモデル・基本操作を構築
   ▼
フェーズ2（UIコンポーネント + 拡張機能）
   │  前提: フェーズ1の描画基盤・コンテナシステム
   │  UIコンポーネントは基本図形の描画パイプライン上に構築
   ▼
フェーズ3（プロジェクト管理 + テンプレート）
   │  前提: フェーズ2のUIコンポーネント
   │  テンプレートにUIコンポーネントを含む可能性がある
   ▼
[PoC-1: HTML/CSS変換品質検証]
   │  Flex/Gridレイアウトを含む代表的なワイヤーフレームの変換検証
   │  結果によりフェーズ4のHTML/CSS変換設計を調整
   ▼
フェーズ4（画像 + HTML/CSSエクスポート）
   │  前提: フェーズ1〜3の全ElementType
   │  全要素タイプの変換ロジックが必要
   ▼
[PoC-2: XAML gap差異の実機検証]
[PoC-3: XAML WrapPanel対応検証]
   │  WPF/WinUI/MAUI/Avalonia間の差異を実機検証
   │  結果によりフェーズ5の各アダプタ設計を調整
   ▼
フェーズ5（XAMLエクスポート）
   │  前提: フェーズ4の変換パイプライン基盤
   │  ExportTargetインターフェースを共有
```

### 2.3 要件定義書フェーズ案からの変更点

| 変更内容 | 理由 |
|---------|------|
| フェーズ1にグループ化（Ctrl+G / Ctrl+Shift+G）を追加 | 複数要素をまとめて移動する操作はMVPでも頻出する基本操作 |
| Gridレイアウトをフェーズ1からフェーズ2に移動 | FreeとFlexだけでMVPは十分機能する。Gridの行・列定義UIが複雑でスコープが膨らむ |
| 遷移図ダイアグラムをフェーズ3に明示的に配置 | ページ間リンク設定が前提であり、プロジェクト管理機能の一部として自然 |
| フェーズ4・5の間にPoC（実現性検証）をゲートとして挿入 | コードエクスポートは最もリスクの高い部分（要件定義書5.2） |
| パフォーマンス最適化をフェーズ横断的な扱いに変更 | テーマ3の方針「PREMATUREな最適化を避ける」に従い、特定フェーズに紐付けない |

## 4. フェーズ2: UIコンポーネント・拡張機能

### 4.1 目標

プリセットUIコンポーネントを使った本格的なワイヤーフレーム作成を可能にする。フェーズ1で先送りされたレイアウト拡張機能を追加し、編集機能を充実させる。

### 4.2 機能スコープ

#### UIコンポーネント（要件定義書 3.4）

11種のUIコンポーネントを一括追加する。内部的には ElementNode の `type: 'uiComponent'` + `componentType` で管理され、描画ロジックの共通部分が多いため、段階的追加のメリットは薄い。

| コンポーネント | 説明 |
|----------------|------|
| ボタン | テキスト付きボタン |
| 入力フィールド | テキスト入力欄、プレースホルダー対応 |
| ドロップダウン | 選択肢リストを持つドロップダウン |
| チェックボックス | ラベル付きチェックボックス |
| ラジオボタン | ラベル付きラジオボタン |
| ナビゲーションバー | 上部ナビゲーションバー |
| サイドバー | 左右のサイドバー |
| リスト | 項目の一覧表示 |
| テーブル | 行・列のテーブル表示 |
| カード | コンテンツをカード形式で表示 |
| モーダル/ダイアログ | オーバーレイ表示されるダイアログ |

#### レイアウト拡張

| 機能 | 詳細 | 先送り元 |
|------|------|---------|
| Gridレイアウト | 行・列の定義、セル配置、セル内の要素ごとの伸縮指定 | フェーズ1から移動 |
| flex-grow的な伸縮 | Flexコンテナ内の子要素の伸縮比率指定 | テーマ3 セクション3.3.3 |

### 4.3 テーマ3/4からの先送り機能の統合

| 機能 | 出典 | 詳細 |
|------|------|------|
| flex-grow | テーマ3 セクション3.3.3 | Flex コンテナ内の子要素に伸縮比率を指定。データモデルへの `flexGrow` プロパティ追加が必要 |
| Grid セル内個別伸縮 | テーマ3 セクション3.4.3 | Grid セル内の要素ごとの伸縮指定。初期実装ではセルサイズに合わせた全体伸縮のみだったものを拡張 |
| ショートカットカスタマイズ | テーマ4 セクション8.3 | 設定UIの追加。実装上はショートカット定義をマッピングテーブルで管理する前提 |

### 4.4 完了基準

- [ ] 11種のUIコンポーネントが配置・編集・カスタマイズできること
- [ ] Gridレイアウトの行・列定義と要素配置が動作すること
- [ ] flex-grow が Flex コンテナで正常に動作すること
- [ ] ショートカットカスタマイズ設定UIが利用できること

### 5.3 遷移図ダイアグラムの位置付け（検討事項B）

遷移図ダイアグラムは**フェーズ3に含める**。

| 観点 | 判断理由 |
|------|---------|
| 前提条件 | ページ間リンク設定が前提であり、同一フェーズで提供するのが自然 |
| 機能的まとまり | クリッカブルプロトタイプとセットで提供した方がプロジェクト管理として完成度が高い |
| 技術的コスト | 既存のKonva.jsキャンバス技術を流用可能。追加の技術スタック不要 |

- [ ] 要素にリンク先ページを設定でき、クリッカブルプロトタイプで遷移を確認できること

フェーズ4開始前に実施する実現性検証。

| 項目 | 内容 |
|------|------|
| **検証内容** | Flex/Gridレイアウトを含む代表的なワイヤーフレームを手動で変換し、CSS出力の品質を検証 |
| **検証観点** | レイアウトの忠実な再現度、CSSプロパティの妥当性、ネストされたコンテナの変換精度 |
| **想定成果物** | 変換ルールの検証結果、課題一覧、フェーズ4への設計フィードバック |
| **判断基準** | 主要なレイアウトパターンが80%以上の忠実度で再現可能であること |
| **リスク対応** | 問題が発覚した場合、HTML/CSSエクスポートの対応範囲を縮小するか、変換方針を見直す |

### 6.4 完了基準

## 8. パフォーマンス最適化方針

### 8.1 基本方針

テーマ3（セクション7.3）で定められた方針に従い、**PREMATUREな最適化を避け、実際にパフォーマンスボトルネックが観測された段階で適切な最適化を導入する**。

パフォーマンス最適化は特定のフェーズに紐付けず、フェーズ横断的な対応として位置付ける。

### 8.2 初期実装に含まれる最適化（フェーズ1）

以下の最適化はフェーズ1の標準実装に含まれるため、追加の実装コストは発生しない。

| 最適化 | 説明 |
|--------|------|
| Vue リアクティブ + vue-konva 自動更新 | 基本的な変更検知と部分再描画 |
| 3レイヤー分離 | グリッド・要素・UI補助の分離による再描画範囲の限定 |
| gridLayer の `listening: false` | イベント処理のオーバーヘッド除去 |

### 8.3 必要時に導入する最適化（フェーズ横断）

以下の最適化は、パフォーマンスボトルネックが観測された時点で導入を検討する。

#### 段階1の最適化

| 最適化 | 説明 | 導入の目安 |
|--------|------|-----------|
| ビューポートカリング | 画面外の要素の描画スキップ | 要素数500超でフレームレート低下時 |
| シェイプキャッシング | 大きなコンテナのオフスクリーンキャッシュ | 子要素20以上のコンテナで再描画コスト増大時 |

#### 段階2の最適化

| 最適化 | 説明 | 導入の目安 |
|--------|------|-----------|
| ドラッグ中のレイヤー分離 | ドラッグ中の要素を overlayLayer に一時移動 | ドラッグ操作時のフレームレート低下時 |
| 仮想化レンダリング | 大量要素時の Vue コンポーネント数削減 | 要素数1000超で初期描画・操作に支障時 |

### 8.4 導入判断基準

テーマ3（セクション7.1）で定めたパフォーマンス目標を基準とする。

| 指標 | 目標値 | 未達時のアクション |
|------|--------|-------------------|
| 通常操作のレスポンス | 60fps（16ms以内） | 段階1の最適化を検討 |
| ドラッグ移動 | 60fps | ドラッグ中のレイヤー分離を検討 |
| 初期描画 | 500ms以内 | 仮想化レンダリングを検討 |
| 快適動作の上限 | 1,000要素/ページ | 段階1/2の最適化を組み合わせて対応 |

## 9. リスク管理

### 9.1 コードエクスポートのリスクとPoC計画

要件定義書（セクション5.2）では、コードエクスポート機能を「本ツール開発全体の中で最もリスクの高い部分」と指摘している。

| リスク | 影響 | 対策 |
|--------|------|------|
| レイアウト再現度が不十分 | エクスポートの実用性が低下 | PoC（3件）で事前検証し、変換ルールを調整 |
| XAMLフレームワーク間のgap差異 | ターゲット別の個別対応が増大 | PoC-2で差異を網羅的に把握し、共通変換基盤で吸収 |
| WrapPanel非対応フレームワーク | Flex wrapの変換が困難 | PoC-3で代替手段を検証。非対応の場合はフォールバック方式を定義 |
| PoC結果による設計変更 | フェーズ4/5のスコープ変更 | PoCの判断基準を事前に定義し、結果に応じて柔軟にスコープを調整 |

**PoC実施のルール**:
- PoCはフェーズ間のゲートとして実施する
- PoCの結果により、後続フェーズのスコープ調整を行う可能性がある
- PoCで重大な問題が発覚した場合、該当エクスポートターゲットの優先度を下げるか、変換方針を見直す

### 9.2 技術的リスク一覧

| リスク | 影響度 | 発生可能性 | 対策 |
|--------|--------|-----------|------|
| Konva.js のパフォーマンス限界 | 高 | 低 | パフォーマンス最適化の段階的導入（セクション8） |
| File System Access API の非対応ブラウザ | 中 | 中 | `<a download>` フォールバック（テーマ2で設計済み） |
| Vuetify コンポーネントのカスタマイズ限界 | 中 | 低 | 自前拡張コンポーネントで補完（横断07で特定済み） |
| IndexedDB のストレージ容量制限 | 低 | 低 | クラッシュリカバリは一時的なバックアップのみ（テーマ2で設計済み） |
| XAML変換の複雑度が想定以上 | 高 | 中 | PoC-2/3で事前検証。対応範囲の限定も選択肢 |

### 9.3 スコープ変更の方針

個人開発プロジェクトとして、以下の方針でスコープを柔軟に管理する。

| 方針 | 詳細 |
|------|------|
| **フェーズ内の優先度調整** | フェーズ内の機能に遅延が生じた場合、優先度の低い機能を次フェーズに移動 |
| **PoC結果による調整** | PoCで重大な問題が発覚した場合、対応範囲を限定または代替方針に切り替え |
| **機能の段階的縮小** | 実装難易度が想定以上の場合、基本機能のみ実装し、拡張は将来対応 |
| **フェーズ統合・分割** | 進捗状況に応じてフェーズの統合（小さいフェーズ同士）や分割（大きいフェーズ）を柔軟に行う |
