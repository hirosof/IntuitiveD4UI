## 1. 概要

本設計書では、Intuitive D4UI におけるキャンバスの描画・レンダリングに関する設計を定める。

### 1.1 本設計書の範囲

| 領域 | 内容 |
|------|------|
| レンダリングパイプライン | ElementNode からCanvas上の描画要素への変換フロー |
| レイアウトエンジン | Free / Flex / Grid レイアウトの自動配置計算ロジック |
| 座標系とトランスフォーム | ローカル座標・グローバル座標の管理、ズーム・パン時の座標変換 |
| キャンバス機能 | キャンバスサイズプリセット、ズーム・パン、グリッド・スナップ、スマートガイド、クリッピング |
| 再描画最適化 | 変更検知・部分再描画、ビューポートカリング、シェイプキャッシング |
| パフォーマンス目標と制約 | 数値目標、ネスト深さ制限、最適化ロードマップ |

### 1.2 前提とする設計書からの入力

本設計書は、以下の設計書で決定された事項を前提とする。

#### テーマ1（アプリケーション基盤）からの前提

- **描画技術**: Canvas 2D（Konva.js + vue-konva）+ DOMオーバーレイ
- **フレームワーク**: Vue 3（Composition API）+ Pinia
- **Konva.jsの役割**: ノードのツリー管理、ヒットテスト、ドラッグ＆ドロップ、変形インタラクション
- **自前実装が必要な領域**: レイアウトエンジン（本設計書で設計）

#### テーマ2（データ設計・永続化）からの前提

- **データモデル**: フラットマップ + 参照ID方式、統一 `ElementNode` 型
- **座標系**: ルート要素はキャンバス絶対座標、子要素は親からの相対座標
- **Zオーダー**: `childrenIds` 配列の順序（先頭が最背面、末尾が最前面）
- **LayoutConfig**: `FreeLayout` / `FlexLayout` / `GridLayout` の3種（コンテナ新規作成時のデフォルトは `FreeLayout`）

## 2. レンダリングパイプライン

### 2.1 パイプライン全体像

データモデルからCanvas上の描画に至るまでのデータフローを以下に示す。

```
[Pinia Store]                    [レイアウトエンジン]              [vue-konva コンポーネント]        [Canvas]
                                  （純粋関数）
elements: Record<id, ElementNode>
        │                                                        Konva.Stage
        │                                                        ├── gridLayer
        ├─── ルート要素 ──────────────────────────────────────────►│   └── グリッド線描画
        │                                                        │
        ├─── 各要素 ─────┬─ Free ─► そのまま x, y 使用 ─────────►├── elementLayer
        │                ├─ Flex ─► Flex計算 → 計算済み座標 ────►│   ├── <WfElement> (再帰)
        │                └─ Grid ─► Grid計算 → 計算済み座標 ───►│   │   ├── <v-rect>
        │                                                        │   │   ├── <v-ellipse>
        │                                                        │   │   ├── <v-text>
        │                                                        │   │   ├── <v-group> (container)
        │                                                        │   │   │   └── <WfElement> ...
        │                                                        │   │   └── <v-group> (group)
        │                                                        │   │       └── <WfElement> ...
        │                                                        │   └── ...
        │                                                        │
        └────────────────────────────────────────────────────────►└── overlayLayer
                                                                      ├── 選択範囲
                                                                      ├── スマートガイド
                                                                      └── 変形ハンドル

[DOMオーバーレイ]
├── テキスト編集用 <textarea>
└── ツールチップ等
```

### 2.2 ElementNode → Konvaノードのマッピング

ElementType ごとに対応する Konva シェイプを以下に定義する。

| ElementType | Konva シェイプ | 備考 |
|-------------|---------------|------|
| `rectangle` | `Konva.Rect` | `cornerRadius` で角丸対応 |
| `ellipse` | `Konva.Ellipse` | `radiusX`, `radiusY` で幅・高さを表現 |
| `line` | `Konva.Line` | `points` プロパティで始点・終点を指定 |
| `arrow` | `Konva.Arrow` | `points` + `pointerAtBeginning` / `pointerAtEnding` |
| `text` | `Konva.Text` | フォント関連プロパティをマッピング |
| `image-placeholder` | `Konva.Group`（`Konva.Rect` + `Konva.Text`） | 矩形 +「IMAGE」等のプレースホルダーテキスト |
| `container` | `Konva.Group`（`Konva.Rect`（背景）+ 子要素） | 背景矩形でコンテナの外観を描画し、子要素を内包 |
| `group` | `Konva.Group`（子要素のみ） | 描画上の固有プロパティなし。子要素をグルーピング |
| `ui-*` | `Konva.Group`（複合シェイプ） | UIコンポーネントごとに矩形・テキスト等の組み合わせ |

#### 共通プロパティのマッピング

ElementNode の共通プロパティは、以下のように Konva ノードの config にマッピングする。

| ElementNode プロパティ | Konva config プロパティ | 備考 |
|----------------------|----------------------|------|
| `x` | `x` | Free / ルート: そのまま、Flex/Grid: レイアウト計算結果を使用 |
| `y` | `y` | 同上 |
| `width` | `width` | 同上 |
| `height` | `height` | 同上 |
| `rotation` | `rotation` | 度単位でそのまま |
| `opacity` | `opacity` | 0.0〜1.0 でそのまま |
| `visible` | `visible` | boolean でそのまま |
| `locked` | `draggable: !locked` | ロック時はドラッグを無効化 |

> **ellipse 固有の変換**: `Konva.Ellipse` は `radiusX` / `radiusY` でサイズを指定するため、ElementNode の `width` / `height` から `radiusX = width / 2`, `radiusY = height / 2` に変換してマッピングする。また、Konva.Ellipse の座標は楕円の中心を指すため、`x` / `y` にもオフセット（`+ width / 2`, `+ height / 2`）を加える。

### 2.3 vue-konva コンポーネント設計

ElementNode の描画には **Vueコンポーネント 1:1 対応方式** を採用する。ElementType ごとに専用の Vue コンポーネントを作成し、vue-konva のコンポーネントをラップする。

#### 2.3.1 コンポーネント構成

```
WfElement.vue          ── 再帰的レンダリングのエントリーポイント
├── WfRectangle.vue    ── rectangle → <v-rect>
├── WfEllipse.vue      ── ellipse → <v-ellipse>
├── WfLine.vue         ── line → <v-line>
├── WfArrow.vue        ── arrow → <v-arrow>
├── WfText.vue         ── text → <v-text>
├── WfImagePlaceholder.vue ── image-placeholder → <v-group>
├── WfContainer.vue    ── container → <v-group>（背景Rect + 子要素再帰）
├── WfGroup.vue        ── group → <v-group>（子要素再帰）
└── WfUIComponent.vue  ── ui-* → <v-group>（複合シェイプ）
```

#### 2.3.2 再帰的レンダリングの仕組み

`WfElement` コンポーネントが `elementId` を受け取り、`type` に応じて適切なコンポーネントに委譲する。コンテナ・グループの場合は `childrenIds` を走査して `WfElement` を再帰的にレンダリングする。

```
<WfElement :element-id="id">
  ├── type に応じてコンポーネントを動的切り替え（component :is）
  └── container / group の場合:
      <v-group :config="groupConfig">
        <v-rect v-if="isContainer" :config="backgroundConfig" />  ← コンテナ背景
        <WfElement
          v-for="childId in element.childrenIds"
          :key="childId"
          :element-id="childId"
        />
      </v-group>
```

#### 2.3.3 採用理由

| 観点 | 1:1 対応方式（採用） | 描画関数方式 |
|------|-------------------|------------|
| 宣言的記法 | Vue のテンプレートで直感的に記述可能 | 手続き的なコード |
| リアクティブ更新 | vue-konva が `:config` の変更を自動検知・反映 | 自前で変更検知・適用が必要 |
| デバッグ | Vue DevTools でコンポーネントツリーを確認可能 | Konvaノードの直接デバッグが必要 |
| パフォーマンス | 数百要素では問題なし。問題発生時はコンポーネント単位でキャッシング導入可能 | オーバーヘッドが小さい |

vue-konva を技術選定した動機が宣言的記法にあり、個人利用ツールの実用的な要素数（数百〜千要素）ではVueコンポーネントのオーバーヘッドが問題になる水準に達しにくいため、方式1を採用する。

#### 2.3.4 Zオーダーの実現

`childrenIds` の配列順で `v-for` レンダリングすることにより、Konva.js の描画順序（後に追加されたものが前面）を利用して自然にZオーダーが実現される。Zオーダーの変更操作は `childrenIds` 配列の要素位置を変更するだけで反映される。

### 2.4 Konva.Layer 分離戦略

Konva.Stage 内のレイヤーを以下の3層に分離する。

```
Konva.Stage
├── Layer 1: gridLayer（グリッドレイヤー）
│   └── グリッド線の描画
│       listening: false（イベント不要）
│
├── Layer 2: elementLayer（要素レイヤー）
│   └── 全ての ElementNode の Konva ノード
│       コンテナ/グループの入れ子構造をそのまま反映
│
└── Layer 3: overlayLayer（UI補助レイヤー）
    ├── 選択範囲の矩形（ラバーバンド）
    ├── スマートガイド（整列ガイドライン）
    ├── 変形ハンドル（Konva.Transformer）
    └── ドロップ先のハイライト
```

#### 分離の設計根拠

| レイヤー | 分離理由 |
|---------|---------|
| gridLayer | 静的な描画。要素の編集時に再描画不要。`listening: false` でイベント処理のオーバーヘッドを除去 |
| elementLayer | ユーザーが操作する全要素を含む。ヒットテスト・ドラッグのイベント処理対象 |
| overlayLayer | 常に要素の上に表示される一時的なUI要素。要素の描画に影響せずに独立して更新可能 |

※ Konva.js の公式推奨は3〜5レイヤー以下。本設計は3レイヤーで推奨範囲内。

### 2.5 DOMオーバーレイの管理

Canvas 上に HTML DOM 要素を重ねて表示する仕組み。主にテキスト編集時に使用する。

#### 2.5.1 使用場面

| 場面 | DOM要素 | 説明 |
|------|---------|------|
| テキスト編集 | `<textarea>` | テキスト要素のダブルクリックで編集モードに入り、Canvas上のテキスト位置にtextareaを重ねて表示 |
| ツールチップ | `<div>` | 要素ホバー時の情報表示（将来的な拡張） |

#### 2.5.2 Canvas座標との同期

DOMオーバーレイの位置は、Canvas の座標変換（ズーム・パン）と同期する必要がある。

```
1. 対象要素のグローバル座標を計算（セクション4.2の変換関数を使用）
2. Konva.Stage の scale（ズーム率）と position（パンオフセット）を考慮
3. DOM要素の CSS transform に反映:
   - left / top: グローバル座標 × ズーム率 + パンオフセット + Stageコンテナのページ上位置
   - transform: scale(ズーム率)
   - transform-origin: top left
```

#### 2.5.3 テキスト編集フロー

```
通常モード: Konva.Text でCanvas上に描画
    │
    ├── ダブルクリック
    ▼
編集モード: Konva.Text を非表示にし、同一位置に <textarea> を表示
    │         <textarea> のスタイルをKonva.Textのプロパティと同期
    │         （フォントサイズ × ズーム率、フォントファミリー、色、配置等）
    │
    ├── Escape / 外部クリック / Enter（設定による）
    ▼
通常モードに復帰: <textarea> の値を ElementNode の props.content に反映
                   <textarea> を非表示にし、Konva.Text を再表示
```

## 3. レイアウトエンジン

### 3.1 全体設計

レイアウトエンジンは **ドメイン層の純粋関数** として実装し、描画技術（Konva.js）に依存しない設計とする。

#### 3.1.1 設計方針

| 方針 | 説明 |
|------|------|
| 純粋関数 | 入力（ElementNode, LayoutConfig, 子要素配列）に対して、出力（計算済み座標・サイズ）を返すのみ。副作用なし |
| 描画技術非依存 | Konva.js への依存を持たない。エクスポート時（テーマ5）にも同じエンジンを再利用可能 |
| ElementNode を書き換えない | 計算結果は別途保持し、ElementNode の `x`, `y`, `width`, `height` は変更しない |

#### 3.1.2 入出力インターフェース

```typescript
/** レイアウト計算の出力（1つの子要素分） */
interface LayoutResult {
  /** 子要素のID */
  elementId: string;
  /** 計算済みX座標（親コンテナからの相対位置） */
  computedX: number;
  /** 計算済みY座標（親コンテナからの相対位置） */
  computedY: number;
  /** 計算済み幅（stretch 等で変更される場合） */
  computedWidth: number;
  /** 計算済み高さ（stretch 等で変更される場合） */
  computedHeight: number;
}

/**
 * レイアウト計算のメインエントリーポイント。
 * container の layout プロパティに基づいてレイアウト種別を判定し、
 * padding を考慮した利用可能領域内で子要素の配置を計算する。
 *
 * @param container - コンテナ要素（layout, padding プロパティを参照する）
 * @param children - 子要素の配列（childrenIds 順、各要素の margin を参照する）
 * @param effectiveSize - 親から与えられたコンテナの実効サイズ。
 *   ルートコンテナの場合や Free レイアウトの子の場合は container.width/height と同値。
 *   Flex/Grid の stretch 等で親からサイズが変更された場合は computedWidth/Height を渡す。
 *   レイアウト計算はこのサイズを基に利用可能領域を算出する。
 */
function calculateLayout(
  container: ElementNode,
  children: ElementNode[],
  effectiveSize: { width: number; height: number },
): LayoutResult[];
```

#### 3.1.3 レイアウト計算と元の座標の関係

| レイアウトモード | x, y の扱い | width, height の扱い |
|----------------|------------|---------------------|
| Free | ElementNode の `x`, `y` をそのまま描画に使用 | ElementNode の `width`, `height` をそのまま使用 |
| Flex / Grid | レイアウト計算結果の `computedX`, `computedY` を描画に使用。ElementNode の `x`, `y` は保持するが描画には使用しない | `alignItems: 'stretch'` やセルサイズ伸縮の場合、`computedWidth` / `computedHeight` を使用。それ以外は ElementNode の値を使用 |

ElementNode の `x`, `y` を保持する理由: ユーザーが Flex/Grid レイアウトから Free レイアウトに切り替えた際に、以前の自由配置位置に復元できるようにするため。

### 3.2 Free レイアウトの処理

Free レイアウト（`layout.type === 'free'`）は自由配置モードであり、子要素の `x`, `y` をそのまま使用する。レイアウト計算は実質的に不要。

> **パディングの扱い**: Free レイアウト時、コンテナに `padding` が設定されていても子要素の座標計算には影響しない。パディングはコンテナの背景描画における視覚的な目安（内側の余白表示）としてのみ機能する。ユーザーが Flex/Grid に切り替えた際に初めてパディングが配置計算に影響する。

```typescript
function calculateFreeLayout(
  container: ElementNode,
  children: ElementNode[],
): LayoutResult[] {
  return children.map(child => ({
    elementId: child.id,
    computedX: child.x,
    computedY: child.y,
    computedWidth: child.width,
    computedHeight: child.height,
  }));
}
```

### 3.3 Flex レイアウトの計算ロジック

#### 3.3.1 計算概要

Flex レイアウトは CSS Flexbox の主要機能をサブセットとして実装する。

```
コンテナの利用可能領域（effectiveSize を基に算出）:
  availableWidth  = effectiveSize.width  - padding.left - padding.right
  availableHeight = effectiveSize.height - padding.top  - padding.bottom

子要素の配置基準点:
  originX = padding.left
  originY = padding.top
```

#### 3.3.2 計算手順

```
1. 子要素のマージンを考慮したサイズの算出
   - 各子要素の占有幅 = child.width + child.margin.left + child.margin.right
   - 各子要素の占有高 = child.height + child.margin.top + child.margin.bottom

2. 主軸方向の合計サイズの算出
   - direction='row':  totalMainSize = Σ(子の占有幅) + gap × (子の数 - 1)
   - direction='column': totalMainSize = Σ(子の占有高) + gap × (子の数 - 1)

3. 折り返し処理（wrap: true の場合）
   - 主軸方向の合計が利用可能サイズを超えた場合、新しい行/列に折り返す
   - 各行/列を独立したグループとして配置計算する

4. justifyContent に基づく主軸方向の配置
   - start:         先頭から詰めて配置
   - center:        余白を均等に両端に分配
   - end:           末尾に寄せて配置
   - space-between: 最初と最後の要素を端に配置し、間の余白を均等分配
   - space-around:  各要素の両側に均等な余白を配置
   - space-evenly:  全ての余白（端含む）を均等に分配

5. alignItems に基づく交差軸方向の配置
   - start:   交差軸の開始位置に揃える
   - center:  交差軸の中央に揃える
   - end:     交差軸の終了位置に揃える
   - stretch: 交差軸方向に引き伸ばす（computedWidth / computedHeight を更新）

6. 各子要素の最終座標を算出
   - computedX = originX + 主軸/交差軸の計算結果 + child.margin.left
   - computedY = originY + 主軸/交差軸の計算結果 + child.margin.top
```

#### 3.3.3 flex-grow 的な伸縮

要件定義書で「子要素のサイズ挙動: 固定サイズの維持、または親コンテナに応じた伸縮を要素ごとに指定可能」と定義されている。これに対応するため、将来的に子要素に `flexGrow` プロパティを追加する拡張パスを残す。

> **実装注記**: 初期実装では固定サイズのみをサポートし、flexGrow 機能はフェーズ2以降で追加する。データモデルへの `flexGrow` プロパティの追加は、実装時にテーマ2の設計書を更新して対応する。

### 3.4 Grid レイアウトの計算ロジック

#### 3.4.1 計算概要

Grid レイアウトは CSS Grid の基本機能をサブセットとして実装する。

```
コンテナの利用可能領域（effectiveSize を基に算出）:
  availableWidth  = effectiveSize.width  - padding.left - padding.right
  availableHeight = effectiveSize.height - padding.top  - padding.bottom

セルサイズの算出:
  cellWidth  = (availableWidth  - columnGap × (columns - 1)) / columns
  cellHeight = (availableHeight - rowGap × (rows - 1)) / rows
  ※ rows=0（自動）の場合: rows = ceil(子要素数 / columns)
```

#### 3.4.2 計算手順

```
1. 行数の決定
   - rows > 0: 指定された行数を使用
   - rows = 0: ceil(子要素数 / columns) で自動算出

2. セルサイズの算出（上記の式）

3. 子要素のセル割り当て
   - childrenIds の順序でセルに左上から順に割り当て
   - セル位置: row = floor(index / columns), col = index % columns
   - rows が明示指定（rows > 0）で子要素数が columns × rows を超過した場合:
     超過した子要素は非表示（描画しない）とし、UIで警告を表示する

4. セル内での子要素配置
   - computedX = padding.left + col × (cellWidth + columnGap) + child.margin.left
   - computedY = padding.top + row × (cellHeight + rowGap) + child.margin.top
   - 子要素のサイズ伸縮: セルサイズに応じた伸縮の指定に従い、
     computedWidth / computedHeight を更新（固定の場合は元のサイズを維持）
```

#### 3.4.3 セル内の伸縮

要件定義書で「セル内の子要素のサイズ挙動: 固定サイズの維持、またはセルサイズに応じた伸縮を要素ごとに指定可能」と定義されている。

> **実装注記**: Flex の flexGrow と同様、初期実装ではセルサイズに合わせた全体伸縮のみをサポートする。要素ごとの伸縮指定はフェーズ2以降で追加する。

### 3.5 計算タイミングと結果保持

レイアウト計算結果の保持には **Vue computed（都度計算）方式** を採用する。

#### 3.5.1 方針

```
[Pinia Store: elementStore]
  state:  elements（フラットマップ）
  getter: getLayoutResults(containerId, effectiveSize): LayoutResult[]
          → container の layout.type に応じて calculateLayout() を呼び出し
          → effectiveSize は親のレイアウト結果から取得（ルートは自身の width/height）
          → Vue の computed として自動キャッシュ・依存追跡
```

#### 3.5.2 採用理由

| 観点 | Vue computed 方式（採用） | 明示的キャッシュ方式 | ElementNode 書き戻し方式 |
|------|------------------------|-------------------|----------------------|
| シンプルさ | 高い。キャッシュ管理は Vue に委譲 | 中。キャッシュの無効化タイミングの管理が必要 | 高い |
| データの明確さ | 「ユーザー指定値」と「計算値」が明確に分離 | 明確に分離 | 区別不能（採用不可） |
| 再計算効率 | Vue の依存追跡で必要な部分のみ再計算 | 手動の差分管理が必要 | — |
| 拡張性 | パフォーマンス問題発生時にキャッシュ方式に移行可能 | — | — |

#### 3.5.3 vue-konva コンポーネントでの利用

コンテナの子要素を描画する際、レイアウトモードに応じて座標を決定する。

```
// WfContainer.vue 内の概念的なロジック
const childConfig = computed(() => {
  if (container.layout.type === 'free') {
    // Free: ElementNode の x, y をそのまま使用
    return { x: child.x, y: child.y, width: child.width, height: child.height };
  } else {
    // Flex / Grid: レイアウト計算結果を使用
    const result = elementStore.getLayoutResults(container.id)
                     .find(r => r.elementId === child.id);
    return { x: result.computedX, y: result.computedY,
             width: result.computedWidth, height: result.computedHeight };
  }
});
```

### 3.6 ネストされたレイアウトの再帰計算

コンテナの入れ子構造において、レイアウト計算は外側から内側へ再帰的に行われる。

#### 3.6.1 計算フロー

```
ルートコンテナ（Flex）
├── 子要素A（通常要素） → ルートコンテナの Flex 計算で配置決定
├── 子コンテナB（Grid）→ ルートコンテナの Flex 計算で B の位置・サイズ決定
│   ├── 孫要素B-1     → コンテナB の Grid 計算で配置決定
│   └── 孫要素B-2     → コンテナB の Grid 計算で配置決定
└── 子要素C（通常要素） → ルートコンテナの Flex 計算で配置決定
```

- 親コンテナのレイアウト計算が子コンテナのサイズ（`computedWidth`, `computedHeight`）を決定する
- 子コンテナのレイアウト計算は、親から与えられた `computedWidth` / `computedHeight` を `effectiveSize` パラメータとして受け取り、これを基に子要素を配置する
- ルートコンテナの場合は、`effectiveSize` に自身の `width` / `height` をそのまま渡す
- Vue の computed の依存追跡により、変更が波及する部分のみ自動的に再計算される

#### 3.6.2 計算順序の保証

Vue の computed は遅延評価であり、参照時に計算される。外側のコンテナの描画時に内側のコンテナのレイアウトが参照されるため、自然に外側→内側の順序で計算が行われる。明示的な計算順序管理は不要。

### 3.7 レイアウトとユーザー操作の整合性

#### 3.7.1 Flex/Grid コンテナ内でのドラッグ移動

Flex/Grid レイアウトのコンテナ内で子要素をドラッグした場合の挙動を以下に定める。

| 操作 | 挙動 |
|------|------|
| コンテナ内でのドラッグ | `childrenIds` 内の順序を変更する（並べ替え）。ドロップ位置に最も近い挿入位置に移動 |
| コンテナ外へのドラッグ | コンテナの子要素から除外し、ドロップ先のコンテナ/ルートに移動 |

> **テーマ4への引き継ぎ**: ドラッグ操作の詳細なインタラクション設計（ドラッグ中のプレビュー表示、挿入位置のハイライト等）はテーマ4（操作・インタラクション設計）で定める。本設計書ではレイアウトエンジン側の「順序変更で再配置」という原則のみを定める。

#### 3.7.2 レイアウトモード切り替え時の挙動

| 切り替え | 挙動 |
|---------|------|
| Free → Flex/Grid | 子要素の `x`, `y` はそのまま保持（復元用）。描画はレイアウト計算結果に従う |
| Flex/Grid → Free | 子要素の `x`, `y` に保持されていた値で自由配置に復帰 |
| Flex → Grid / Grid → Flex | 計算済み座標が切り替え後のレイアウトで即座に再計算される |

## 4. 座標系とトランスフォーム

### 4.1 座標系の定義

テーマ2で定められた座標系を描画の観点から整理する。

| 座標系 | 基準点 | 使用場面 |
|--------|--------|---------|
| キャンバス座標 | キャンバス左上が原点 (0, 0) | ルート要素の `x`, `y`。キャンバスサイズの定義 |
| ローカル座標 | 親要素の左上が原点 (0, 0) | 子要素の `x`, `y`（親からの相対位置） |
| ビューポート座標 | ブラウザ表示領域の左上が原点 | マウスイベントの座標。DOMオーバーレイの配置 |

### 4.2 ローカル座標 ↔ グローバル座標の変換

任意の要素のキャンバス上の絶対座標（グローバル座標）を取得するために、祖先要素の座標を累積する変換関数を設計する。

```typescript
/**
 * 要素のグローバル座標（キャンバス上の絶対座標）を算出する。
 * 祖先を辿り、各階層の座標（レイアウト計算結果を考慮）を累積する。
 */
function getGlobalPosition(
  elementId: string,
  elements: Record<string, ElementNode>,
  getLayoutResult: (containerId: string, childId: string) => LayoutResult | null,
): { x: number; y: number } {
  let globalX = 0;
  let globalY = 0;
  let currentId: string | null = elementId;

  while (currentId !== null) {
    const element = elements[currentId];
    const parent = element.parentId ? elements[element.parentId] : null;

    if (parent && parent.layout && parent.layout.type !== 'free') {
      // Flex/Grid コンテナの子: レイアウト計算結果を使用
      const result = getLayoutResult(parent.id, currentId);
      globalX += result ? result.computedX : element.x;
      globalY += result ? result.computedY : element.y;
    } else {
      // Free / ルート要素: ElementNode の x, y を使用
      globalX += element.x;
      globalY += element.y;
    }

    currentId = element.parentId;
  }

  return { x: globalX, y: globalY };
}
```

> **実装注記**: Konva.js は `node.getAbsolutePosition()` メソッドを提供しており、Konva.Group のネスト構造が正しく構成されていれば、このメソッドでもグローバル座標を取得可能。上記関数はレイアウトエンジン側（描画技術非依存）での座標変換用として設計している。

### 4.3 ズーム・パン時の座標変換

ズーム・パンは Konva.Stage の `scaleX`, `scaleY`（ズーム率）と `x`, `y`（パンオフセット）で管理する。

#### 4.3.1 座標変換の式

```
ビューポート座標 → キャンバス座標:
  canvasX = (viewportX - stage.x()) / stage.scaleX()
  canvasY = (viewportY - stage.y()) / stage.scaleY()

キャンバス座標 → ビューポート座標:
  viewportX = canvasX * stage.scaleX() + stage.x()
  viewportY = canvasY * stage.scaleY() + stage.y()
```

#### 4.3.2 Konva.js での実装

Konva.Stage は `getPointerPosition()`（ビューポート座標）と `getRelativePointerPosition()`（Stage のローカル座標 = キャンバス座標）を提供する。マウスイベント時にはこれらのメソッドを使用して座標変換を行う。

| 最大倍率 | 800% |
| デフォルト倍率 | 100% |

#### 5.2.2 ズーム操作

| 操作 | 挙動 |
|------|------|
| Ctrl + マウスホイール | マウスカーソル位置を中心に連続的なズーム |
| UIボタン（+/−） | 所定ステップで段階ズーム |
| ズーム率直接入力 | 任意の倍率を数値で指定 |
| フィット表示 | ページ全体がビューポートに収まる倍率に自動調整 |

**UIボタンのステップ**: 10%, 25%, 50%, 75%, 100%, 150%, 200%, 400%, 800%

> **実装注記**: 100%未満と100%以上でステップ幅が非対称（例: 200%→400%間に300%がない）である点は、実装時に操作感を検証し、必要に応じてステップを追加・調整する。

#### 5.2.3 ズームの実装方針

```
// マウスカーソル位置を中心としたズーム
1. マウスのビューポート座標を取得
2. 現在のズーム率から新しいズーム率を算出
3. マウス位置がズーム前後で同じキャンバス座標を指すように、
   Stage の position を補正:
   newStageX = pointerX - (pointerX - oldStageX) * (newScale / oldScale)
   newStageY = pointerY - (pointerY - oldStageY) * (newScale / oldScale)
4. Stage の scale と position を更新
```

| 操作 | 挙動 |
|------|------|
| スペース + ドラッグ | キャンバスをパン（手のひらツール的操作） |
| マウスホイール（Ctrlなし） | 垂直方向のパン |
| Shift + マウスホイール | 水平方向のパン |
| 中クリック + ドラッグ | キャンバスをパン |

### 5.3 グリッド表示・スナップ

#### 5.3.1 グリッド設定

| 項目 | 値 |
|------|-----|
| デフォルト間隔 | 8px |
| 設定可能範囲 | 4px〜100px |
| 初期表示状態 | OFF（非表示） |
| 表示切り替え | ON/OFF トグル |

#### 5.3.2 グリッドの描画

- gridLayer（セクション2.4）に描画する
- キャンバス領域内のみにグリッド線を描画する
- グリッド線の色は薄いグレー（例: `#E0E0E0`）で、主線（一定間隔ごと）をやや濃く表示する

#### 5.3.3 グリッドの視覚的密度制御

ズームアウト時にグリッド線が密になりすぎることを防ぐため、画面上の表示密度を自動制御する。

```
方針:
  画面上でのグリッド線の間隔 = グリッド間隔 × ズーム率

  この間隔が 8px 未満になった場合、表示間隔を段階的に倍加する:
  - 表示間隔 = グリッド間隔 × 2^n  （n は間隔が 8px 以上になる最小の自然数）

例（グリッド間隔 8px の場合）:
  ズーム 100%: 画面上 8px間隔  → そのまま表示
  ズーム  50%: 画面上 4px間隔  → 16px間隔（2倍）で表示
  ズーム  25%: 画面上 2px間隔  → 32px間隔（4倍）で表示
  ズーム  10%: 画面上 0.8px間隔 → 64px間隔（8倍）で表示
```

#### 5.3.4 スナップ設定

| 項目 | 値 |
|------|-----|
| グリッドスナップ | ON/OFF 切替可能、デフォルト OFF |
| スマートガイドスナップ | ON/OFF 切替可能、デフォルト ON |
| スナップ吸着閾値 | 画面上 5px 以内に接近した場合にスナップ |

#### 5.3.5 グリッドスナップのロジック

```
要素の移動/リサイズ時:
1. 移動先/リサイズ先の座標を算出
2. 最も近いグリッド交点までの距離を算出
3. 距離 ≤ 吸着閾値（5px / ズーム率 = キャンバス座標での閾値）の場合:
   → グリッド交点にスナップ
4. 距離 > 吸着閾値 の場合:
   → スナップなし（自由配置）
```

### 5.4 スマートガイド（整列ガイドライン）

要素の移動時に他の要素との位置関係を視覚的に表示し、整列を補助する機能。

#### 5.4.1 判定対象

移動中の要素と同一階層（同じ親を持つ）の他要素について、以下の位置関係を比較する。ルート要素同士（`parentId: null` 同士）も同一階層として判定対象に含まれる。

> **将来拡張**: キャンバス境界（端線・中心線）との整列ガイドは、初期実装では対象外とし、将来的な拡張として検討する。

| 判定項目 | 説明 |
|---------|------|
| 左辺同士 | 移動中の要素の左辺と、他要素の左辺が一致 |
| 右辺同士 | 移動中の要素の右辺と、他要素の右辺が一致 |
| 上辺同士 | 移動中の要素の上辺と、他要素の上辺が一致 |
| 下辺同士 | 移動中の要素の下辺と、他要素の下辺が一致 |
| 水平中心同士 | 移動中の要素の水平中心と、他要素の水平中心が一致 |
| 垂直中心同士 | 移動中の要素の垂直中心と、他要素の垂直中心が一致 |
| 等間隔ガイド | 3つ以上の要素が水平/垂直方向に等間隔に並ぶ場合 |

#### 5.4.2 表示ロジック

```
1. 移動中の要素の6つの基準線（左辺、右辺、上辺、下辺、水平中心、垂直中心）を算出
2. 同一階層の他要素の6つの基準線と比較
3. 差分 ≤ スナップ吸着閾値（5px / ズーム率）の場合:
   a. overlayLayer にガイドライン（青色の実線）を描画
   b. 移動中の要素をガイドラインの位置にスナップ
4. 移動終了時にガイドラインを消去
```

#### 5.4.3 ガイドラインの視覚的表現

- **色**: 青色（例: `#0066FF`）
- **線種**: 実線、太さ 1px（ズームに関わらず一定の画面上太さ）
- **端部**: キャンバス端まで延長、または関連する要素間のみ（実装時に調整）

### 5.5 子要素のクリッピング

コンテナの境界における子要素の表示制御。

#### 5.5.1 overflow プロパティ

ElementNode のコンテナ関連プロパティに `overflow` を追加する（テーマ2の設計書を併せて更新する）。

```typescript
interface ElementNode {
  // ... 既存フィールド ...

  /** コンテナの子要素オーバーフロー設定（type が container の場合のみ有効） */
  overflow?: 'visible' | 'clip';
}
```

| 値 | 挙動 | CSS相当 |
|----|------|---------|
| `visible`（デフォルト） | 子要素がコンテナ境界をはみ出しても全て描画 | `overflow: visible` |
| `clip` | コンテナ境界で子要素をクリッピング | `overflow: hidden` |

#### 5.5.2 実装方針

`overflow: 'clip'` の場合、Konva.Group の `clipFunc` プロパティを使用してクリッピング領域を設定する。

```
// clip 時の Konva.Group config
{
  clipFunc: (ctx) => {
    ctx.rect(0, 0, container.width, container.height);
  }
}
```

#### 5.5.3 デフォルト値の理由

デフォルトを `visible` とする理由:
- ワイヤーフレーム作成中は試行錯誤的に要素を配置するため、はみ出しが許容される方が操作しやすい
- 「クリップして表示」はレイアウトの最終確認時に必要になる機能であり、オプションとして提供する

## 6. 再描画最適化

### 6.1 変更検知と部分再描画

Vue のリアクティブシステムと vue-konva の自動更新機能を活用し、変更があった要素のみを効率的に再描画する。

#### 6.1.1 基本方針

```
Pinia Store（elements）
    │ リアクティブな変更
    ▼
Vue の依存追跡（computed / watch）
    │ 変更検知
    ▼
vue-konva コンポーネント
    │ :config プロパティの変更を検知
    ▼
Konva ノードの attrs 更新
    │
    ▼
Konva.Layer.batchDraw()（自動呼び出し）
    │ 該当レイヤーのみ再描画
    ▼
Canvas 更新
```

- vue-konva は `:config` にバインドされたオブジェクトの変更を自動検知し、対応する Konva ノードのプロパティを更新する
- Konva.js は変更があったレイヤーのみを再描画する（レイヤー単位の部分再描画）
- 個々の Konva ノードの変更は、そのノードが属するレイヤー全体の再描画をトリガーする

#### 6.1.2 最適化のポイント

| 最適化 | 説明 |
|--------|------|
| computed の活用 | Konva の config オブジェクトを computed で生成し、依存データが変わらない限り再生成しない |
| リアクティブの粒度 | ElementNode のプロパティ単位で変更を検知するため、無関係な要素の再計算は発生しない |
| レイヤー分離 | グリッドの再描画と要素の再描画を分離し、不要なレイヤーの再描画を防止 |

### 6.2 ビューポートカリング

ビューポート外の要素の描画をスキップして、パフォーマンスを向上させる手法。

#### 6.2.1 方針

Konva.js にはビューポートカリングが組み込まれていないため、必要に応じて自前で実装する。

**初期実装では見送り**、パフォーマンスが問題になった段階で導入する。ただし、コンポーネント設計時にカリング導入を容易にする構造にしておく。

#### 6.2.2 導入時の設計

```
1. ビューポートの矩形を算出
   - Stage の position と scale からキャンバス座標でのビューポート範囲を逆算

2. 各ルート要素のバウンディングボックスとビューポートの交差判定
   - 交差しない要素: node.visible(false) で描画をスキップ
   - 交差する要素: node.visible(true)

3. コンテナの最適化
   - コンテナのバウンディングボックスがビューポートと交差しない場合、
     子要素全体をスキップ（内部の個別チェック不要）

4. 更新タイミング
   - ズーム・パン操作時にビューポート矩形を再計算
   - デバウンスを適用して頻繁な再計算を防止
```

### 6.3 シェイプキャッシング

Konva.js の `cache()` メソッドを使用して、複雑なグループをオフスクリーンキャンバスにキャッシュし、再描画コストを削減する手法。

#### 6.3.1 適用方針

**初期実装では見送り**、以下の条件でパフォーマンスが問題になった段階で導入を検討する。

| 適用候補 | 条件 |
|---------|------|
| 子要素が多いコンテナ | 子要素数が一定以上（例: 20以上）で、かつコンテナ内の要素が編集されていない場合 |
| ロックされた要素グループ | `locked: true` の要素は変更されないため、キャッシュ効果が高い |

#### 6.3.2 注意事項

- キャッシュされた要素を編集する場合は、キャッシュの無効化（`clearCache()`）が必要
- キャッシュはメモリを消費するため、適用範囲を限定する
- テキストのアンチエイリアシング品質がキャッシュにより低下する場合がある

### 6.4 ドラッグ中の最適化

#### 6.4.1 基本方針

初期実装では Konva.js の標準ドラッグ処理に任せる。パフォーマンスが問題になった段階で以下の最適化を検討する。

#### 6.4.2 将来の最適化候補

| 手法 | 説明 |
|------|------|
| ドラッグ中のレイヤー分離 | ドラッグ中の要素を elementLayer から overlayLayer に一時移動し、elementLayer の再描画を抑制 |
| `dragmove` のスロットリング | dragmove イベントの処理頻度を requestAnimationFrame に同期 |
| 周辺要素の簡略描画 | ドラッグ中は周辺要素を簡略化（枠線のみ）して描画コストを削減 |

## 7. パフォーマンス目標と制約

### 7.1 パフォーマンス数値目標

個人利用のワイヤーフレームツールとして、以下のパフォーマンス目標を設定する。

| 指標 | 目標値 | 計測条件 |
|------|--------|----------|
| 通常操作のレスポンス | 60fps（16ms以内） | 要素数 500 以下、ネスト 3 階層以内 |
| ドラッグ移動 | 60fps | 単一要素のドラッグ、子要素 100 以下のコンテナ含む |
| レイアウト再計算 | 50ms 以内 | Flex コンテナ内の子要素 20 個の再配置 |
| 初期描画（ページ読み込み） | 500ms 以内 | 要素数 500 のページの初回描画 |
| ズーム・パン | 60fps | 要素数 500 のキャンバスでのスムーズなズーム・パン |
| 快適動作の上限要素数 | 1,000 要素/ページ | フレーム落ちなく操作可能な上限の目安 |
| 限界動作の上限要素数 | 3,000 要素/ページ | レスポンスが低下するが操作可能な上限の目安 |

> **補足**: 個人利用のワイヤーフレームツールで、1ページあたりの実用的な要素数は数百程度を想定する。UIコンポーネント（内部的に複数 Konva シェイプで構成される場合がある）も考慮して余裕を持った目標とした。

### 7.2 ネスト深さの制限

コンテナの入れ子（ネスト）深さに対して、以下の制限を設ける。

| 種別 | 値 | 扱い |
|------|-----|------|
| 推奨上限 | 8 階層 | UIで注意表示（「ネストが深くなっています」）。操作は制限しない |
| システム上限 | 16 階層 | これ以上のネストは操作を拒否する。エラーメッセージを表示 |

#### 7.2.1 制限値の根拠

| 観点 | 説明 |
|------|------|
| 実用性 | HTML の実用的な DOM 構造でも、意味のあるネストは通常 5〜8 階層程度 |
| 表現力 | 16 階層あれば十分に複雑な構造を表現可能 |
| 技術的制約 | JavaScript のコールスタック制限（通常 10,000 以上）には遠く及ばない。レイアウト計算の再帰深度としても十分 |
| 精度 | ネスト深さに伴う座標変換の累積による浮動小数点精度低下を防止 |

#### 7.2.2 制限の適用箇所

- コンテナへの要素ドロップ時にネスト深さをチェックする
- ペースト操作時にネスト深さをチェックする
- テンプレート展開時にネスト深さをチェックする

### 7.3 最適化ロードマップ

パフォーマンス最適化を段階的に導入する計画。

| フェーズ | 最適化 | 説明 |
|---------|--------|------|
| **フェーズ1（初期実装）** | Vue リアクティブ + vue-konva 自動更新 | 基本的な変更検知と部分再描画。追加実装なし |
| | 3レイヤー分離 | グリッド・要素・UI補助の分離による再描画範囲の限定 |
| | gridLayer の `listening: false` | イベント処理のオーバーヘッド除去 |
| **フェーズ2（必要時）** | ビューポートカリング | 画面外の要素の描画スキップ |
| | シェイプキャッシング | 大きなコンテナのオフスクリーンキャッシュ |
| **フェーズ3（必要時）** | ドラッグ中のレイヤー分離 | ドラッグ操作の高速化 |
| | 仮想化レンダリング | 大量要素時の Vue コンポーネント数削減 |

> **方針**: PREMATURE な最適化を避け、実際にパフォーマンスボトルネックが観測された段階で適切な最適化を導入する。
